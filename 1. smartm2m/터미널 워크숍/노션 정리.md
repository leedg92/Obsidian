### TSS 활성화 집중기간 운영 계획

- TSS 사용률 현황: 전체 35% 처리 중, BCT/PNC/PNIT에서 50% 이상 운영
- 집중기간 운영 계획 수립 중 - 터미널과 PPA의 역할 분리하여 각 터미널에 공유 예정
- 운송사 TSS 사용 독려 및 발생 오류에 대한 최우선 개선 처리 요청
- 9개 터미널의 개선 속도 차이로 운송사들에게 양치기 소년 같은 상황 발생 중
- 팀장들의 팀원 시간 할애 환경 조성 및 사장단의 인지와 지원 필요

### TSS 주요 개선 필요 사항

**우선순위 높은 항목:**

- 4번 항목: 황기사 요구로 민원이 많이 발생하는 부분
- 컴바인 처리: 총중량 초과하지 않는 범위 내에서 처리하도록 개선 필요. 무조건 막는 것은 TSS 사용 기피 요인
- 반출 취소 기능 보완
- 작년과 비교 시 개선사항 처리가 거의 진행되지 않은 상태

**시스템 개선 필요:**

- 구르보도 처리기 보완
- 반출 취소 시 반입 터미널 데이터 자동 삭제 로직 추가 (현재 데이터 누적으로 두 번째 진입 시 ECO 발생)
- 터미널별 방입 제한 문제는 우선순위 낮음

### 터미널 인센티브 제도

- 대상: 9개 터미널 중 6개 터미널에 인센티브 지급
- 산정 방식: 매 분기별 등수를 4분기 합산하여 평균 순위 산출
- 지급액: 1등 5천만원, 2등~6등 차등 지급 (3백만원~5백만원)
- 체인포털 내 TSS 인센티브 메뉴 개발 중, 6월까지 완료 목표
- 2분기부터 사장단 간담회에서 성과 공유 예정

### 포타이(Port-i) 시스템 현황

**데이터 정확성 문제:**

- 터미널 홈페이지 정보와 포타이 정보 불일치로 많은 시간과 인력 소모
- 선사와 운송사가 어느 정보가 맞는지 문의하는 상황 발생
- 데이터 비교 검증 화면 개발 완료, 터미널별 개별 방문하여 현장에서 함께 검증 예정

**주요 개발 계획:**

- IIS 컨테이너 마이그레이션: 많이 쓰는 기능들을 포타이로 이전 및 재정비
- ETW 데이터 연계: VBS에서도 중요한 데이터, 가능한 터미널부터 진행
- 오픈 API: 표준화된 데이터를 터미널에 제공

**올해 개발 예정 주요 기능:**

1. 글로벌 AIS 연동: 캡플로와 협의하여 글로벌 선박 위치 추적 가능하도록 확장
2. AI 기반 연결 예측: 선박 위치와 속도 분석하여 연결 가능 여부 사전 예측
3. 선사 메신저 기능: 포타이/체인포털 가입자 간 검증된 자료 교환
4. 컨테이너 위치 추적: 전 터미널에서 특정 컨테이너 위치 확인 가능

**선박 스케줄 기능:**

- 전배, 셔틀 발생 시 터미널 입장에서 다양한 상황 확인 가능
- 자동 알림 기능으로 업무 효율성 향상
- 선석별 시뮬레이션 기능 제공
- 가상 선박 스케줄 시뮬레이션 활용 가능

### VBS (차량예약시스템) 추진 계획

**현재 상황:**

- 현재 프로세스는 기사 수동 예약 중심으로 실효성 부족
- 진입 5-10분 전 코피노 전송으로 사전 준비 불가능
- 국토부 주관 도로안전 관련 회의에서 VBS 언급, 올해 시작 필요

**새로운 접근 방식:**

- 기사 수동 예약에서 데이터 기반 예측 구조로 전환
- GPS 의무 활성화 후 AI 기반 ETA 산출 (기사 위치, 운행 패턴, 터미널 혼잡도, 교통 상황 등 고려)
- 운송사가 코피노 전달 후 최소 1시간 전 전송 필요

**컨테이너 유형별 프로세스:**

_일반 컨테이너:_

- AI 기반 ETA 자동 산정 후 터미널에 정보 제공
- 기사에게 시간 재확인 절차 포함 (초기 테스트 단계에서는 생략 고려)
- 터미널은 확정 예약 정보를 신뢰하고 야드 운영에 반영

_냉동 컨테이너:_

- 사전 예약제로 운영 (하루 또는 3일 전 예약)
- 반출/반입 예정 컨테이너 리스트 업로드 시 시스템이 자동으로 최적 시간 제안
- 양하 전이면 야드 스태킹 계획에 활용, 양하 완료 시 야드 상황 고려한 예약 유도
- 플러그 이슈로 인해 온도 단위별로 단계적 실행 필요 (센시티브 제외하고 냉동화물만 우선)

_직반출입:_

- ETW 시간에 맞춰 트럭 도착 필요, 예약제 대상 아님
- 터미널에서 정확한 시간 안내하여 바로 처리 가능하도록

_반입:_

- 예약 개념이 필요한지 재검토 필요
- AI 기반 ETA 산정하여 터미널에 정보 제공
- 노쇼 관련 정책 수립 필요

**도입 방식:**

- 단계적 도입 검토: 냉동 → 직반출 → 일반컨 순차 진행 또는 동시 진행
- 특정 터미널 선도 도입 vs 전 터미널 동시 도입에 대한 의견 수렴 필요
- 상반기 방향성 수립, 하반기 개발, 4분기 테스트 목표
- 용역 발주 완료 상태

**VBS 목표:**

1. 리핸들링 최소화 (양하 순서대로 반출 순서로 스태킹)
2. 11시-3시 급픽크 타임 분산하여 터미널 부하 감소

**터미널별 의견:**

_PNIT:_

- PL 활용하여 양하 시부터 리핸들링 최소화
- 일반 화물 턴타임 15% 미만으로 짧아 예약 실익 검토 필요
- 냉동컨테이너 전배 물량 많아 예약 필요성 높음 (MSC 물량)
- 천일운송사와 사전 리스트 기반 운영 중
- MSC는 자체 시스템으로 관리, 매달 냉동컨테이너 그룹아웃 시간 사후정산

_BCT:_

- 오버사이즈 컨테이너 사전 신청 시스템 운영 중, 웹 접수로 개선 예정

_HJNC:_

- 11시-해질녘 차량 집중 문제 해결이 목표
- 슬롯 시스템 정책적 지원 필요
- 차량 큐잉 상황의 GPS 정보 제공 필요 (기사들 간 순서 분쟁 해결용)
- 개인정보 문제 있으나 공익적 목적으로 위치정보 제공 요청

_DCT:_

- 슬롯 개념을 AI 에이전트로 관리하는 것 제안
- 시간당 처리 용량 초과 시 턴타임 비례 증가 문제
- 선적 리핸들링 98%, 반출 리핸들링 45% - 차이는 사전 정보 단계 수
- 세관 정보 기반 AI 개발 중이나 터미널 데이터만으로는 한계, BPA 데이터 활용 희망
- POC 대상 확대 가능

_HPNT:_

- VBS 효용 분석 필요
- 예약 정보 있으면 사전 작업으로 리핸들링 최소화 가능
- 냉동컨테이너 플라우스 관리 중요
- 야드 점유율 75%, ETW 개선 요청

### 전배 물량 TSS 셔틀 허용

- PNIT 한정으로 현재 진행 중, 최근 전배 물량 감소
- 시스템 체계 마련 완료, KNF 연계 작업 완료
- 전산 조치 불필요, 운영 관점 조치만 필요
- 각 터미널 내부적으로 체계 수립 필요

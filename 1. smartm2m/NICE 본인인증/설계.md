# NICE 통합인증 연동 구현 설계

## 1. 개요

### 1.1 목적
allcone 앱에서 NICE 통합인증 서비스를 연동하기 위한 백엔드 구현 설계

### 1.2 연동 구조
```
allcone 클라이언트 ↔ allcone 백엔드 ↔ 골디락스 서버 ↔ NICE 서버
```
- allcone 백엔드는 API 키 등 민감 정보를 클라이언트에 노출하지 않기 위한 중계 역할
- 골디락스 서버가 NICE 서버와의 직접 통신 담당

### 1.3 참고 API (골디락스 제공)

| API명 | URI | Method |
|-------|-----|--------|
| 접근 토큰 발급 | `/ido/intc/v1.0/auth/token` | POST |
| 인증 URL 요청 | `/ido/intc/v1.0/auth/url` | POST |
| 인증 결과 요청 | `/ido/intc/v1.0/auth/result` | POST |

---

## 2. 전체 흐름

```
[1]  클라이언트 → 백엔드 : 인증 URL 요청
[2]  백엔드 : access_token 유효성 확인 → 없거나 만료 시 발급
[3]  백엔드 : request_no 생성
[4]  백엔드 → 골디락스 서버 : auth/url 호출 (return_url, close_url에 request_no 포함)
[5]  골디락스 서버 → 백엔드 : auth_url + transaction_id 반환
[6]  백엔드 : request_no + transaction_id DB INSERT
[7]  백엔드 → 클라이언트 : auth_url 반환
[8]  클라이언트 : auth_url로 인증창(WebView/팝업) 오픈
[9]  사용자 : NICE 인증창에서 인증 진행

     ── 인증 완료 시 ──
[10] NICE → return_url 호출 (request_no + web_transaction_id 포함)
[11] 백엔드 : access_token 유효성 재확인 → 만료 시 재발급
[12] 백엔드 : request_no로 DB 조회 → transaction_id 획득
[13] 백엔드 → 골디락스 서버 : auth/result 호출
[14] 백엔드 : enc_data 복호화 + integrity_value 무결성 검증
[15] 백엔드 : DB row 삭제
[16] 백엔드 → 클라이언트 : 인증 결과 반환 (name, CI/DI, 생년월일 등)

     ── 인증창 닫기 시 ──
[10] NICE → close_url 호출 (request_no 포함)
[11] 백엔드 : request_no로 DB row 즉시 삭제

     ── 앱 강제종료 등 ──
     새벽 배치로 expired_at 초과 row 정리 (보험)
```

---

## 3. 백엔드 API 설계

### 3.1 인증 URL 요청
```
GET /api/nice/auth-url
```

**처리 흐름**
1. access_token 유효성 확인 (만료 시 재발급)
2. request_no 생성
3. return_url, close_url에 request_no 쿼리파라미터 포함하여 body 구성
4. 골디락스 서버 `/auth/url` 호출
5. DB INSERT (request_no, transaction_id, created_at, expired_at)
6. auth_url 반환

**응답**
```json
{
  "auth_url": "https://..."
}
```

### 3.2 인증 결과 처리 (return_url)
```
GET /api/nice/result?request_no={request_no}&web_transaction_id={web_transaction_id}
```

**처리 흐름**
1. **access_token 유효성 재확인** (만료 시 재발급)
2. request_no로 DB 조회 → transaction_id 획득
3. 골디락스 서버 `/auth/result` 호출
4. enc_data 복호화 (AES/GCM/NoPadding, 256bit)
5. integrity_value 무결성 검증 (HmacSHA256)
6. DB row 삭제
7. 클라이언트에 인증 결과 전달

> **샘플코드 바꿔야됨** : 제공된 샘플코드의 `decryptNiceResult`에는 토큰 만료 체크 로직이 없음.
> url 요청 시점에 토큰이 유효하더라도 사용자가 인증창에서 시간을 소요하는 사이 토큰이 만료될 수 있으므로
> result 요청 시점에도 반드시 토큰 유효성을 재확인하는 로직을 추가해야 함.

### 3.3 인증창 닫기 처리 (close_url)
```
GET /api/nice/close?request_no={request_no}
```

**처리 흐름**
1. request_no로 DB row 즉시 삭제

## 4. 데이터 관리 설계

### 4.1 access_token 관리

access_token은 API 호출 시 Authorization 헤더에 포함되는 이용기관 인증값으로,
유효 여부만 검증되며 request_no/transaction_id 조합과는 독립적이다.
이중화 환경에서 서버별로 독립 관리해도 무방하다.

#### Case 1. 인메모리 캐시 (이게 제일 나을것같음)
- 서버별로 독립적으로 발급 및 캐싱
- 기존 allcone 백엔드의 `EslipPopupDomainService` Semaphore 락 방식 레퍼런스 참고
- 서버 재시작 시 재발급 발생하나 오류 없음

```
[발급 조건]
- 서버 최초 기동 시
- 토큰 만료 시 (만료 1분 전 갱신, expires_in 값 기준)

[만료 체크 적용 시점]
- auth/url 호출 전 (url 요청 처리 시)
- auth/result 호출 전 (result 요청 처리 시) ← 반드시 포함
  → url 요청 후 사용자 인증 소요 시간 동안 토큰이 만료될 수 있음

[인메모리 저장 대상]
- access_token, ticket, iterators, expires_in
  → 토큰 발급 응답값 전체를 캐싱 (ticket, iterators는 복호화 시 필요)

[동시성 처리]
- Semaphore(1) 락으로 중복 발급 방지
```

#### Case 2. DB 저장 + 스케줄링 갱신
- DB에 access_token 및 응답값 전체 저장
- `@Scheduled`로 만료 전 주기적 갱신
- 이중화 환경에서 두 서버가 동일 토큰 공유 가능
- 인메모리 대비 구현 복잡도 증가

```sql
CREATE TABLE nice_access_token (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    token       VARCHAR(1000) NOT NULL,
    ticket      VARCHAR(200)  NOT NULL,
    iterators   INT           NOT NULL,
    expires_at  DATETIME      NOT NULL,
    updated_at  DATETIME      NOT NULL
);
```

### 4.2 request_no / transaction_id 관리

#### Case 1. DB저장 (이게 제일 나을것같음)
`auth/url` 호출 시 생성/수신한 값으로, `auth/result` 호출 시 반드시 동일한 값이 필요하다.
이중화 환경에서 url 요청과 result 요청이 서로 다른 서버로 라우팅될 수 있어
인메모리 저장 불가, 공유 저장소(ex. DB)에 보관해야 한다.

#### 테이블 설계

```sql
CREATE TABLE nice_auth_request (
    request_no      VARCHAR(50)  PRIMARY KEY,   -- UUID 기반 생성, 중복 가능성 없음
    transaction_id  VARCHAR(100) NOT NULL,
    created_at      DATETIME     NOT NULL,
    expired_at      DATETIME     NOT NULL        -- created_at + N분
);
```

#### 데이터 생명주기

```
[INSERT] auth/url 요청 시
[SELECT] auth/result 요청 시 request_no 기준 조회
[DELETE - 즉시] return_url 처리 완료 시 → 정상 인증 완료 케이스
[DELETE - 즉시] close_url 호출 시       → 사용자가 인증창 직접 닫은 케이스
[DELETE - 배치] 새벽 배치               → 앱 강제종료 등 찌꺼기 정리 (보험)
```

```java
// 새벽 배치 - 보험용
@Scheduled(cron = "0 0 3 * * *") // 매일 새벽 3시
public void cleanExpiredAuthRequest() {
    niceAuthRequestRepository.deleteByExpiredAtBefore(LocalDateTime.now());
}
```

#### Case 2. JEUS 세션 클러스터링 활용

JEUS 클러스터링(`192.168.100.58`, `192.168.100.59`)이 이미 세션 공유로 구성되어 있다면,
access_token, request_no, transaction_id 모두 세션으로 관리 가능하며 DB 저장이 불필요하다.

```
[확인 필요] JEUS 클러스터링 세션 공유 설정 여부
            → 설정되어 있다면 샘플 코드(HttpSession 방식) 그대로 활용 가능
```

## 5. 복호화 로직 설계

### 5.1 키 유도 (PBKDF2)
```
PBEKeySpec(ticket, transactionId, iterators, 512)
→ Base64UrlEncoding(withoutPadding)
→ keyString (512bit)
  - 앞 32byte  → 대칭키 (AES 암복호화)
  - 48~80byte → HMAC 키 (무결성 검증)
```
- `ticket`, `iterators` : `/auth/token` 응답값 (인메모리에서 조회)
- `transactionId` : `/auth/url` 응답값 (DB에서 조회)

### 5.2 무결성 검증 (HmacSHA256)
```
HMAC-SHA256(encData, hmacKey) → Base64UrlEncoding
→ integrity_value와 비교
→ 불일치 시 위변조로 처리 후 예외 발생
```

### 5.3 복호화 (AES/GCM/NoPadding)
```
Base64UrlDecode(encData)
→ 앞 16byte : IV
→ 나머지    : 암호화된 인증 결과
AES/GCM/NoPadding(key, IV) → 복호화 → JSON 파싱
```

## 6. 사전 준비 사항

| 항목                           | 내용                                          | 담당     |
| ---------------------------- | ------------------------------------------- | ------ |
| 서버 IP 등록                     | allcone 백엔드 Outbound IP → 골디락스사 전달          | 이동근 대리 |
| client_id / client_secret 발급 | 골디락스사 통해 발급 확인                              | ?      |
| 방화벽 오픈                       | 골디락스 서버로의 Outbound 허용 여부 확인                 | 이동근 대리 |
| JEUS 세션 클러스터링 여부             | 세션 공유 설정 여부 확인                              | 이동근 대리 |
| return_url / close_url 처리 방식 | 앱에서 WebView 내 URL 호출 방식 및 인증 완료 결과 전달 방식 협의 | 윤혜림 사원 |

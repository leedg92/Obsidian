## 1. Phase 1 wrap + ① 4 방향 gateway 첫 구현

**5/7 공유 시점 잔여 작업 (Phase 2-A 마감 + 외부 IO 컴포넌트 + chainportal 회원 보강) 모두 wrap + 5/14 에 bctrans inbound bypass mirror 페이즈 추가 wrap. 테스트베드 진입 직전 상태.**

- 신 allcone 호환 마감 (Phase 2-A) — URL/method 21/21 매칭 / alias↔legacy 응답 envelope 일치 / docs 신규 endpoint 응답 필드 규칙 정렬 / EventCode outbox INSERT 12종 (신규 3종 포함)
- 외부 IO 컴포넌트 (handler 모듈) 본격 구현 — bctrans core 의 외부 호출 8 카테고리 30+ 지점 전수 handler 위임 전환 (② bctrans outbound)
- chainportal handler 회원 보강 — 신 allcone 이 호출할 임시 대역 endpoint 32개 미러링 (auth 5 + sign 6 + user 18 + alarm_config 2 + by-truck 1)
- **bctrans inbound bypass mirror (신규, 5/14 wrap) — 외부 시스템(BaaS/터미널/AAS 운송사/BPA) 호출 8 prefix 를 handler 가 받아 core 로 raw HTTP forward. ① bctrans inbound 방향 첫 구현**
- 단방향 위반 (handler → core/allcone import) 최종 0건 확인

**작업 페이스 메모**: chainportal이 회원 디비를 소유하고 API를 열어주는 쪽으로 확정 후 신 allcone 측에서 호출할 회원관련 chainportal API의 완성을 기다리기 전 handler 가 chainportal DB 직접 접속하는 임시 대역으로 자체 완결. chainportal API 완성 시 handler 내부 forwarding 교체만으로 전환 가능 (endpoint 영구 유지, wire format 동결).

**4 방향 gateway 룰 (5/14 확정, docs/msa-architecture-draft.md §2.2)**: handler 가 외부 시스템 IO 의 양방향 단일점. ① bctrans inbound / ② bctrans outbound (5/8 완료) / ③ 신 allcone inbound / ④ 신 allcone outbound (구체 범위는 신 allcone repo 전수 조사 follow-up). 모듈 간 도메인 호출(allcone↔core 등)은 handler 미경유 직접 호출.

## 2. bctrans 측 작업 진척

**검토자 결과**: CRITICAL 0, HIGH 1→해결, MEDIUM 5 보류 (호환 무관)

### 외부 IO 페이즈 (2026-05-08 wrap)

handler 모듈 어댑터 신규 8종 + core 호출 위임 8종.

| 어댑터 (handler 신설) | core 위임 전환 |
| --- | --- |
| HandlerClient 공통 인프라 (envelope DTO + HandlerCallException + handlerRestTemplate Bean + handler.url) | BChainService / VbsBChainService — BchainHandlerClient |
| KTNet movement 어댑터 (WebClient + 10s timeout fire-and-forget) | TerminalRequestClient*Service — handler 위임 (외부 호출만, bctrans loop-back 은 WebClient 유지) |
| TMS 어댑터 (vendor 라우팅 + fire-and-forget WebClient) | TssGroupOrderPushService / PushTruckerService — TmsHandlerClient (truckerId→vendor 매핑은 handler 내부) |
| BChain 어댑터 (ITT/VBS 채널 분기 + RestTemplate 동기 + code 검사) | MovementAsyncService — handler 위임 (vendor 조회는 core 유지) |
| Terminal 어댑터 (apiKey 헤더 + 동기 RestTemplate 이식) | FCMNotificationService — handler 위임 (Firebase SDK 직접 호출 제거, dryRun 호환만 유지) |
| chainportal 어댑터 (truckNo→userId 임시 DB 직접 접속, Bean 교체 가능) | AASAdapter / BlockChainAdapter — handler 위임 (raw JSON forward + status/body 분기, 외부 진입점 URL 무변경) |
| FCM 어댑터 (Firebase Admin SDK + 단건/멀티캐스트 + Retryable 이식) | SendNotificationToTruckerAdapter — TruckerHandlerClient (TSS callback raw forward) |
| AAS 인증 어댑터 (validate/callback raw body forward + 5xx/timeout 매핑) | — |

**위임 결과**: bctrans core 는 외부 시스템 직접 호출 0건. handler 가 영구 IO gateway 로 자리 잡음 (신 allcone 도 외부 호출 시 handler 경유 강제).

### chainportal handler 회원 보강 (2026-05-12 wrap)

`feature/memberTableChange` 의 회원 4 패키지를 handler/chainportal 에 미러링. 신 allcone 이 회원 도메인 호출 시 handler 가 임시 대역.

| 카테고리 | endpoint | 비고 |
| --- | --- | --- |
| auth (DeviceAuth + NiceAuth) | 5 | NICE 본인인증 외부 호출 포함 |
| sign (SignIn/SignUp + DeviceToken) | 6 | FCM 토큰 등록은 신 allcone 책임으로 분리 |
| user (UserInfo + UserTruck) | 18 | 차량등록증 multipart 업로드 + 대표차량 / `users/by-truck` UserTruckController 통합 |
| alarm_config | 2 | 옵션 C — 터미널 의존성 + 자동 초기화 제거 (신 allcone 이 사용자별 알람/터미널 매핑 자체 관리) |

**공통 인프라 추가**: egov+Oracle datasource+crypt(Text/CI)+SMS+ApiResponse / NICE config + apiRestTemplate Bean / 차량등록증용 multipart 50MB 한도 / typeHandler 2개 XML resultMap 명시.

**누계**: 9 commit / ~81 파일 / ~6,500+ 라인 / 단방향 위반 0건.

### bctrans inbound bypass mirror (2026-05-14 wrap)

핸들러 모듈에 CoreBypassController + CoreBypassService + CoreBypassException + coreRestTemplate Bean 신설. 외부 시스템이 호출하는 bctrans inbound 8 prefix 를 handler 가 받아 core 인스턴스로 raw HTTP forward (URL/method/응답 status/body/headers 모두 무변경).

| 카테고리 | mirror prefix | 외부 호출자 |
| --- | --- | --- |
| BaaS invoke alarm | `/vbs/invoke/**`, `/itt/invoke/**`, `/odcy/invoke/**` | 자사 BaaS |
| 터미널 이벤트 | `/terminal/gatein/**` | 터미널 시스템 |
| AAS 인증 운송사 | `/tss/open-api/**`, `/tss/group/invoke/**` | TSS 운영자 측 운송사 |
| BPA apiKey | `/tss/getInOutTerminal/**`, `/terminal/cycapacity/**` | BPA |

**검토 결과 (3-way 1차 + 2차)**: CRITICAL 1 (tss/group/invoke 누락) → fix / HIGH 1 (RFC 7230 §6.1 hop-by-hop 헤더 미차단) → fix / 재검토에서 BPA 2 endpoint 누락 발견 → fix / 502 응답 schema 정렬 (core ErrorResponse 와 wire-compatible JSON 으로) → fix / MEDIUM 5 (DoS / SSRF / localhost default / multipart trap door / 비표준 HTTP method) → 자체 follow-up (활성화 전 처리)

**누계**: 6 commit / 8 파일 / ~390 라인. 동작: 본 commit 만으로는 receive-ready 상태 (실제 트래픽 전환은 LB 라우팅 변경 별도 작업).

**외부 시스템 영향**: URL/method/응답 status/body/headers/schema 모두 무변경. 502 BAD_GATEWAY 도 core 의 ErrorResponse 와 wire-compatible JSON 으로 정렬. **외부 통지 불필요**.

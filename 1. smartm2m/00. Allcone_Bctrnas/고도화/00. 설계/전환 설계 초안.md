# MSA 아키텍처 초안

> 작성일: 2026-04-06
> 상태: 초안 (회의 진행하면서 변경 가능)
> 배경: 모노리스(단일 WAR) → MSA(도커 컨테이너) 전환 검토

## 전환 동기

- TSS 관리를 다른 회사에 이관 예정 (데이터 정제 후 전송 수준의 작은 역할)
- 현재 단일 WAR에서 Spring 프로필(allcone/bctrans)로 분기 배포 중
- allcone ↔ bctrans 간 강한 커플링 존재 (allcone→bctrans 64건 직접 참조)

## 구조

```
              [앱]
                ↕
            [allcone]              앱 프록시 + 회원관리
                ↕
        [transport-core]           공통 운송이벤트 DB/처리
           ↕         ↕
         [vbs]     [tss]           각 운송 유형별 전처리 (코피노 등)
           ↕         ↕
      [transport-handler]          운송 관련 외부 단일 진입/진출점
        ↕    ↕    ↕    ↕
   터미널  블록체인  다른회사  기타외부
```

## 서비스 목록 (각각 독립 도커 컨테이너)

| 서비스                   | 성격           | 역할                                                                | 안 하는 일                  |
| --------------------- | ------------ | ----------------------------------------------------------------- | ----------------------- |
| **allcone**           | 앱 프록시 + 자체DB | 앱 인증, 회원CRUD, FCM, 앱↔백엔드 라우팅. 자체 DB(회원/FCM 등) 소유                  | 운송 로직, 운송 DB 직접 접근      |
| **transport-core**    | 중앙 데이터       | 공통 운송이벤트 DB 적재/전처리/상태관리                                           | 외부 통신, 앱 통신, 앱 전용 DB 접근 |
| **vbs**               | 전처리 모듈       | VBS 코피노 처리, 비즈니스 규칙 적용 후 transport-core 호출. 전처리용 자체 DB 소유 가능      | 공통 운송 DB 직접 접근          |
| **tss**               | 전처리 모듈       | TSS/그룹오더 코피노 처리, 비즈니스 규칙 적용 후 transport-core 호출. 전처리용 자체 DB 소유 가능 | 공통 운송 DB 직접 접근          |
| **transport-handler** | 외부 프록시       | 외부 인증, 프로토콜 변환, vbs/tss로 라우팅                                      | 비즈니스 로직                 |

## 통신 흐름

### 앱 요청 (앱 → 외부)
```
앱 → allcone → transport-core → vbs or tss → transport-handler → 외부
```

### 외부 이벤트 (외부 → 앱)
```
외부 → transport-handler → vbs or tss → transport-core → allcone → 앱(FCM)
```

### 다른 회사 (TSS 관리)
```
다른 회사 → transport-handler → tss → transport-core
```
- 다른 회사는 transport-core API 스펙에 맞게 데이터를 정제하여 전송
- 앱 푸시나 DB 적재가 필요하면 transport-core를 통해서만 가능

## 확정 사항

- **서비스 5개**, 각각 독립 도커 컨테이너
- **공통 jar 없음**, 코드 중복 허용 (각 모듈이 자기 enum/VO/유틸 소유)
- **모듈 간 통신은 API만** (OpenAPI 스펙 기반)
- **transport-handler**: 운송 관련 외부 단일 진입/진출점
- **allcone**: 앱 프록시 + 회원 등 앱 전용 데이터. 외부 소통 필요 시 직접 가능 (transport-handler 안 거침)
- **테이블 소유 원칙: 각 서비스가 자기 테이블만 접근** (물리 DB 분리 여부는 미확정)
  - allcone: 앱 전용 테이블(회원, FCM 등) 소유 → allcone만 접근
  - transport-core: 공통 운송 테이블 소유 → transport-core만 접근
  - vbs/tss: 각자 전처리용 테이블 소유 가능 → 해당 서비스만 접근
- **다른 서비스 소유 테이블에 직접 접근 금지** → 반드시 API 호출
  - 예: allcone에서 운송 데이터 필요 → transport-core API 호출
  - 예: 단순 회원 조회 → allcone 자체 테이블에서 직접 처리 (transport-core 거치지 않음)
- **vbs/tss는 서로의 DB에 절대 접근 불가**
- 다른 회사(TSS)는 데이터 정제 후 전송만 담당, 핵심 로직은 우리가 소유

## 미확정 사항 (회의에서 결정)

### 물리 DB 분리 여부
- 테이블 소유권은 확정 (각 서비스가 자기 테이블만 접근)
- 물리적으로 DB를 나눌지, 같은 DB에서 테이블 권한만 분리할지는 미확정
- 코피노 처리가 공통화되면 → vbs/tss는 테이블 없이 순수 전처리 모듈로 경량화 가능
- vbs를 transport-core에 합칠 가능성도 있음 (코피노 처리만 다른 경우)

### sync/async 경계
- 현재 모노리스는 1홉, MSA는 최대 4홉 → 레이턴시 증가
- 어디를 async로 끊을지 정해야 함
- 예시: transport-core → allcone (푸시) 구간을 async로 하면 DB 적재는 확실히 하면서 푸시 실패에 안전

### 이중화
- transport-core가 SPOF (Single Point of Failure) → replicas 필수
- 도커 기반이므로 구현은 어렵지 않으나 설계 시 고려 필요

### allcone 외부 소통 방식
- allcone이 외부와 직접 소통할지, transport-handler를 경유할지

### 코피노 처리 위치
- 현재: vbs/tss 각 모듈이 담당
- 공통화 가능하면 위치 재검토

### 커플링 현황
- allcone → bctrans: **64건** (22개 파일) — 주로 bctrans.common.enums(40건), bctrans.vbs(21건)
- bctrans → allcone: **5건** (4개 파일) — 약한 커플링
- allcone → common: 72건, bctrans → common: 79건 — 양쪽 균등

### 문제점
- `bctrans.common.enums`가 잘못된 위치 (allcone 18파일, bctrans 67파일, common 10파일에서 참조)
- allcone이 bctrans.vbs의 entity/mapper/repository를 직접 접근 (순수 프록시가 아님)
- invoke 패키지가 allcone/bctrans 양쪽 브릿지 역할

> 상세 분석 데이터는 이 문서 작성 시점의 코드 기준이며, MSA 전환 시 참고용입니다.

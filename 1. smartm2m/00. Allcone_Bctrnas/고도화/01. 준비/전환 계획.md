# MSA 점진적 전환 계획

> 작성일: 2026-04-08 (2026-04-09 개정)
> 상태: 초안 (논의 진행하면서 변경 가능)
> 관련: [MSA 아키텍처 초안](msa-architecture-draft.md)

## 전제

- 점진적 Phase 전략
- **프로필은 allcone / bctrans 2개만 유지** (도커 전환 전까지)
  - allcone-api: allcone
  - bctrans-api: transport-core + vbs + tss + transport-handler
- 공통 코드는 각 패키지가 복사해 중복 소유 (공유 jar 없음)
- **패키지 간 소통은 HTTP API 호출만** (직접 함수호출 금지)
- 각 패키지 안에 자기만의 common(enum/VO/유틸) 보유

## 최종 목표 구조

```
              [앱]
                ↕
            [allcone]              앱 프록시 + 회원관리
                ↕
        [transport-core]           공통 운송이벤트 DB/처리
           ↕         ↕
         [vbs]     [tss]           각 운송 유형별 전처리 (코피노 등)
           ↕         ↕
      [transport-handler]          운송 관련 외부 단일 진입/진출점
        ↕    ↕    ↕    ↕
   터미널  블록체인  TMS  기타외부
```

## 현재 패키지 → 목표 소속 매핑

| 현재 패키지      | 소속                     | 비고                          |
| ----------- | ---------------------- | --------------------------- |
| allcone     | → allcone              |                             |
| bctrans     | → bctrans              |                             |
| gps         | → allcone              | 독립 추가 패키지, allcone과만 소통     |
| insa        | → allcone              | 회원 문자인증 (가입/아이디찾기/비밀번호 재설정) |
| invoke      | → bctrans (handler 후보) | Vbs/Itt/Odcy/Notice 알람 수신   |
| dataLinkage | → bctrans (core 후보)    | TSS/VBS 운송 데이터 연동           |
| healthCheck | → 양쪽 복사                |                             |
| common      | → 분배 후 소멸              | 하위 패키지별 분배 (아래 참고)          |
| oldbctrans  | → 확인 후 폐기 여부 결정        | Old~ 레거시, 사용 여부 체크 필요       |

### common 하위 패키지 분배

| common 하위 | 소속 | 판단 근거 |
|---|---|---|
| tss/ (entity/mapper/repository/service/vo/convert) | → bctrans.tss | TSS 도메인 통째로 |
| vbs/ (vo) | → bctrans.vbs | VBS 전용 |
| terminal/ (entity/mapper/repository/service/web/dto/vo) | → bctrans (handler 후보) | 터미널 연동 |
| trans/ (mapper/repository/service/vo) | → bctrans (core 후보) | 공통 운송 |
| transportPopup/ (entity/mapper/repository/service/vo) | → bctrans (core 후보) | 운송 팝업 |
| push/ (dto/service/vo/web) | → allcone | FCM 푸시 |
| constant/ + typehandler | → 양쪽 복사 | enum/상수 |
| config/ | → 양쪽 복사 | 설정 |
| response/ | → 양쪽 복사 | API 응답 래핑 |
| support/ (annotation/async/bchain/crypt/exception/filter) | → 확인 후 분배 | bchain→handler, 나머지는 양쪽 복사 가능성 |
| util/ | → 확인 후 분배 | 양쪽 복사 가능성 |

## Phase 1-1: common → allcone/bctrans에 분배 (common 소멸)

### 목표
common 패키지를 allcone과 bctrans에 소유권 기준으로 분배하여 완전히 제거

### 작업 내용
- 위 매핑표에 따라 common 하위 패키지를 각 소속으로 이동
- gps, insa 패키지를 allcone으로 이동
- support/, util/ 내부 확인 후 분배
- import 경로 전부 수정
- common 패키지 삭제

### 완료 기준
- common 패키지가 존재하지 않음
- 각 패키지가 자기 common(enum/VO/유틸)을 내부에 보유

---

## Phase 1-2: allcone ↔ bctrans 커플링 해소

### 목표
allcone과 bctrans 간 직접 참조를 전부 HTTP API 호출로 전환

### 작업 내용
- allcone → bctrans 직접 참조 64건을 HTTP API 호출로 전환
- bctrans → allcone 역방향 참조 5건도 HTTP API 호출로 전환
- 트랜잭션 경계 검토

### 완료 기준
- allcone과 bctrans가 서로의 코드를 직접 참조하지 않음
- 모든 소통이 HTTP API로만 이루어짐

---

## Phase 1-3: bctrans 내 handler 패키지 분리

### 목표
외부 통신 로직을 handler 패키지로 집중

### 작업 내용
- 터미널, 블록체인, 컨테인어스 등 외부 API 호출 로직을 handler로 이동
- invoke 패키지를 handler로 흡수
- vbs/tss → handler → 외부 호출 방향 확립

### 완료 기준
- 외부와 직접 통신하는 코드가 handler에만 존재

---

## Phase 1-4: bctrans 내 vbs ↔ tss 크로스 참조 제거

### 목표
vbs와 tss가 서로 독립적으로 동작하도록 분리

### 작업 내용
- vbs ↔ tss 간 크로스 참조 제거
- 공통으로 쓰는 코드는 각자 복사

### 완료 기준
- vbs와 tss가 서로의 코드를 참조하지 않음

---

## Phase 1-5: transport-core 패키지 도입

### 목표
공통 운송 로직을 core 패키지로 정리하여 5개 패키지 경계 완성

### 작업 내용
- transport-core 패키지 생성
- dataLinkage, common.trans, common.transportPopup 등 공통 운송 로직을 core로 집중
- allcone → core → tss/vbs 호출 경로 확립

### 완료 기준
- bctrans 내 4개 패키지(core/vbs/tss/handler) 경계 확립
- 패키지 간 직접 참조 없음

---

## Phase 3: 멀티 모듈 전환

### 목표
하나의 WAR에서 모듈별 독립 빌드로 전환

### 작업 내용
- Maven 멀티 모듈 프로젝트로 전환
- allcone / core / tss / vbs / handler 각각 독립 모듈
- 모듈별 독립 빌드 및 배포 가능하도록 구성

### 완료 기준
- 각 모듈이 독립적으로 빌드되어 별도 산출물이 나옴
- 모듈 간 소스코드 의존성 없음

---

## Phase 4: 도커 컨테이너 전환

### 목표
프로필 기반 배포에서 컨테이너 기반 배포로 전환

### 작업 내용
- 모듈별 Dockerfile 작성
- 컨테이너 오케스트레이션 구성
- bctrans 내부 패키지 간 호출을 HTTP 통신으로 전환
- 서비스 간 네트워크, 헬스체크, 모니터링 구성
- transport-core 이중화 (SPOF 방지)

### 완료 기준
- 5개 서비스가 각각 독립 컨테이너로 기동
- 서비스 간 API 통신 정상 동작

---

## Phase 5: DB 분석 + 공통 스키마 설계 + core 실체화

### 목표
transport-core가 소유할 공통 운송 스키마를 설계하고, core를 실제 데이터 처리 서비스로 전환

### 작업 내용
- TSS/VBS 테이블 DDL 확보 및 컬럼 전수 조사
- 공통 운송 스키마 설계 (TSS 플랫 구조 vs VBS 정규화 구조 통합)
- 데이터 마이그레이션
- core가 공통 스키마를 직접 소유/접근하도록 전환
- tss/vbs에서 공통 DB 접근 로직 제거 → core API 호출로 전환

### 완료 기준
- core가 공통 운송 DB를 소유하고 직접 처리
- tss/vbs는 전처리만 담당, 공통 DB 접근 불가
- **최종 목표 구조 달성**

## 참고: 주요 걸림돌

- common.tss에 TSS 도메인 핵심(mapper/repository/service/entity)이 통째로 존재
- allcone → bctrans 직접 참조 약 64건 (양방향 순환 의존 포함)
- TSS 플랫 1테이블(85컬럼) vs VBS 정규화 4테이블 → 스키마 통합 시 재설계 필요
- TSS 단건과 그룹 TSS의 프로세스 흐름이 본질적으로 다름
- common 패키지의 60%는 특정 모듈에 편중, 진짜 전체 공유는 40% 수준

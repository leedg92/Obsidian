# MSA 점진적 전환 계획

> 작성일: 2026-04-08
> 상태: 초안 (논의 진행하면서 변경 가능)
> 관련: [MSA 아키텍처 초안](msa-architecture-draft.md)

## 전제

- 올해 안에 전체 전환은 비현실적 → 점진적 Phase 전략
- DB 변경 없이 코드 분리를 먼저 진행
- 공통 코드는 각 모듈이 복사해 중복 소유(시간이 지날수록 모듈에 맞는 공통코드로 변할거니까)
- 현재 서버 2대 이중화, 프로필 기반(allcone/bctrans) 배포 구조를 활용(확인 필요)
- 각 Phase가 끝날 때마다 눈에 보이는 성과가 있도록 구성

## 최종 목표 구조

```
              [앱]
                ↕
            [allcone]              앱 프록시 + 회원관리
                ↕
        [transport-core]           공통 운송이벤트 DB/처리
           ↕         ↕
         [vbs]     [tss]           각 운송 유형별 전처리 (코피노 등)
           ↕         ↕
      [transport-handler]          운송 관련 외부 단일 진입/진출점
        ↕    ↕    ↕    ↕
   터미널  블록체인  TMS  기타외부
```

- 서비스 5개, 각각 독립 컨테이너
- 모듈 간 통신은 API만
- transport-core가 공통 운송 DB 소유
- vbs/tss는 전처리만 담당, 서로의 DB 접근 불가

---

## Phase 1: transport-handler 패키지 도입

### 목표
모노리스 안에서 외부 통신 로직을 handler 패키지로 모아 단일 진입/진출점 확보

### 작업 내용
- 터미널, 블록체인, 컨테인어스, 기타 외부 API 호출 로직을 handler 패키지로 이동
- invoke alarm 수신 로직도 handler 경유하도록 정리
- vbs/tss → handler → 외부, 외부 → handler → vbs/tss 호출 방향 확립
- 프로필 간 HTTP 호출로 연결 (기존 invoke alarm 패턴과 동일)
- 나중에 컨테이너 분리 시 URL만 변경하면 되도록 설계

### 완료 기준
- 외부와 직접 통신하는 코드가 handler 패키지에만 존재
- vbs/tss가 외부 API를 직접 호출하지 않음

### 성과
- 외부 연동 변경 시 handler만 수정하면 됨
- 이후 분리 시 그대로 떼어낼 수 있는 준비 완료

---

## Phase 2: allcone 분리 + transport-core 프록시 도입

### 목표
allcone을 독립 서비스로 떼어내고, core를 프록시로 미리 도입하여 최종 구조의 호출 경로 확보

### 작업 내용

#### 2-1. common 패키지 정리 (allcone 분리를 위한 선행)
- common.tss에 박힌 도메인 핵심(mapper/repository/service/entity)을 tss 소유로 이동
- common 내 enum/VO를 소유권 기준으로 분배
  - allcone 전용 → allcone으로
  - tss 전용 → tss로
  - vbs 전용 → vbs로
  - 여러 모듈이 쓰는 것 → 각 모듈에 복사
- 사용하지 않는 import, 미사용 코드 정리

#### 2-2. allcone ↔ bctrans 커플링 해소
- allcone → bctrans 직접 DB 접근(약 68건)을 API 호출로 전환
- bctrans → allcone 역방향 참조(약 4건)도 API 호출로 전환
- 트랜잭션 경계 검토: 기존에 하나의 트랜잭션으로 묶였던 로직이 서비스 분리 시 어떻게 되는지 확인

#### 2-3. transport-core 프록시 도입
- core 패키지 생성, 현 단계에서는 프록시 역할만 수행
- allcone → core → tss/vbs 호출 경로 확립
- 향후 공통 DB를 소유할 예정임을 문서/주석으로 명시

#### 2-4. 프로필 분리 (allcone + core)
- allcone, core를 독립 프로필로 등록
- @Profile, ComponentScan 범위 제한
- 기존 서버에 프로필 추가 배포

### 완료 기준
- allcone이 bctrans의 코드를 직접 참조하지 않음
- allcone → core → tss/vbs 호출 경로로 동작
- allcone, core가 독립 프로필로 기동 가능

### 성과
- 모노리스에서 가장 큰 덩어리가 분리됨
- 프로필 2개 → 4개(allcone, core, bctrans, handler)로 서비스가 눈에 보이게 늘어남
- 컨테인어스 이관의 기반 마련

---

## Phase 3: tss / vbs 분리

### 목표
bctrans 덩어리에서 tss와 vbs를 각각 독립 서비스로 분리

### 작업 내용

#### 3-1. tss ↔ vbs 경계 명확화
- tss와 vbs 간 크로스 참조 제거
- 공통으로 쓰는 코드는 각자 복사
- 그룹 TSS 로직은 tss에 귀속

#### 3-2. 프로필 분리 (tss + vbs)
- bctrans 프로필을 tss, vbs 프로필로 분리
- 각 프로필이 자기 컨트롤러/서비스만 스캔하도록 제한

### 완료 기준
- tss와 vbs가 서로의 코드를 참조하지 않음
- tss, vbs가 각각 독립 프로필로 기동 가능
- 5개 프로필 완성: allcone, core, tss, vbs, handler

### 성과
- 최종 목표의 서비스 5개가 프로필 수준에서 완성
- 컨테인어스에 tss 프로필만 넘길 수 있는 상태
- 각 서비스의 책임 범위가 명확해짐

---

## Phase 4: 멀티 모듈 전환

### 목표
하나의 WAR에서 모듈별 독립 빌드로 전환

### 작업 내용
- Maven 멀티 모듈 프로젝트로 전환
- allcone / core / tss / vbs / handler 각각 독립 모듈
- 각 모듈이 자기 코드만 포함 (common 모듈 없음, 필요한 코드는 복사)
- 모듈별 독립 빌드 및 배포 가능하도록 구성

### 완료 기준
- 각 모듈이 독립적으로 빌드되어 별도 산출물(WAR/JAR)이 나옴
- 모듈 간 소스코드 의존성 없음 (API 통신만)

### 성과
- 코드 수준에서 완전한 분리 달성
- 각 모듈의 빌드/배포가 독립적
- 도커 전환의 직전 단계

---

## Phase 5: 도커 컨테이너 전환

### 목표
프로필 기반 배포에서 컨테이너 기반 배포로 전환

### 작업 내용
- 모듈별 Dockerfile 작성
- 컨테이너 오케스트레이션 구성 (Docker Compose 또는 K8s)
- 서비스 간 네트워크 설정, 서비스 디스커버리
- 헬스체크, 로깅, 모니터링 구성
- transport-core 이중화 (SPOF 방지)

### 완료 기준
- 5개 서비스가 각각 독립 컨테이너로 기동
- 서비스 간 API 통신 정상 동작
- 이중화 및 장애 대응 구성 완료

### 성과
- 물리적 MSA 전환 완료
- 서비스별 독립 스케일링 가능
- 인프라 수준의 현대화

---

## Phase 6: DB 분석 + 공통 스키마 설계

### 목표
TSS/VBS의 테이블 구조를 분석하고, transport-core가 소유할 공통 운송 스키마를 설계

### 작업 내용
- TSS/VBS 전체 테이블 DDL 확보 및 컬럼 전수 조사
- 컬럼 간 의미 매핑표 작성 (이름만 다르고 같은 데이터 식별)
- 3분류: TSS에만 있는 것, VBS에만 있는 것, 공통인 것
- 공통 운송 스키마 설계
  - TSS 플랫 구조(85컬럼)와 VBS 정규화 구조(4테이블)의 통합 방안
  - 상태 관리 방식 통합 (TSS 직접 저장 vs VBS 이력 기반)
- 데이터 마이그레이션 계획 수립 및 실행

### 완료 기준
- 공통 운송 스키마가 확정되고 DB에 반영됨
- 기존 데이터가 새 스키마로 마이그레이션 완료
- 기존 서비스 동작에 영향 없음 (병행 운영 기간 포함)

### 성과
- transport-core가 소유할 데이터의 실체가 확정됨
- 이후 core 실체화의 기반 완성

---

## Phase 7: transport-core 실체화

### 목표
프록시였던 core를 실제 공통 운송 데이터 처리 서비스로 전환

### 작업 내용
- core가 공통 스키마 테이블을 직접 소유/접근하도록 전환
- 공통 운송이벤트 적재, 상태관리, 전처리 로직을 core에 구현
- tss/vbs에서 공통 DB 접근 로직 제거, core API 호출로 전환
- tss/vbs는 순수 전처리 모듈로 경량화

### 완료 기준
- core가 공통 운송 DB를 소유하고, 직접 처리
- tss/vbs는 전처리만 담당, 공통 DB 접근 불가
- allcone → core → tss/vbs → handler 최종 흐름 동작

### 성과
- **최종 목표 구조 달성**
- 각 서비스의 책임이 명확히 분리
- 컨테인어스에 tss만 깔끔하게 이관 가능

## Phase 요약

| Phase | 내용 | DB 변경 | 핵심 성과 |
|-------|------|---------|----------|
| 1 | handler 패키지 도입 | 없음 | 외부 통신 단일화 |
| 2 | allcone 분리 + core 프록시 | 없음 | 첫 번째 서비스 독립 |
| 3 | tss / vbs 분리 | 없음 | 5개 프로필 완성 |

| 6 | DB 분석 + 공통 스키마 설계 | 있음 | core 데이터 확정 |
| 7 | core 실체화 | 있음 | 최종 구조 완성 |

## 참고: 주요 걸림돌

- common.tss에 TSS 도메인 핵심(mapper/repository/service/entity)이 통째로 존재
- allcone → bctrans 직접 DB 접근 약 68건 (양방향 순환 의존 포함)
- TSS 플랫 1테이블(85컬럼) vs VBS 정규화 4테이블 → 스키마 통합 시 재설계 필요
- TSS 단건과 그룹 TSS의 프로세스 흐름이 본질적으로 다름
- common 패키지의 60%는 특정 모듈에 편중, 진짜 전체 공유는 40% 수준

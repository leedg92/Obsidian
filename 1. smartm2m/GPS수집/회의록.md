# GPS 수집 시스템 아키텍처

## 시스템 구성

```
App → Bctrans/Allcone backend (바이패스) → Proxy (리버스프록시) → Cloud 서버 (검증 + Kafka produce) → Kafka → ClickHouse (consume + 적재)
```

| 컴포넌트        | 위치      | 역할                                 |
| ----------- | ------- | ---------------------------------- |
| App (모바일)   | 클라이언트   | GPS 수집, 필터링(isMoving==0 제외), 배치 전송 |
| Bctrans/Allcone backend | BPA IDC | 바이패스만                              |
| Proxy       | 프록시 서버  | nginx 리버스프록시 설정, 클라우드로 라우팅         |
| Cloud 서버    | 클라우드    | 검증(한반도 경계) + Kafka produce          |
| Kafka       | 클라우드    | 메시지 큐 (토픽: gps-points 등)           |
| ClickHouse  | 클라우드    | Kafka consume + GPS 데이터 적재 + 설정 메타데이터 테이블 |

---

## 앱 (모바일)

- `isMoving == 0`인 포인트는 **제외하고 전송** (정지 상태 데이터 불필요)
- **운송 오더가 존재할 때만** GPS 데이터 수집
  - 단, 백그라운드로 계속 수집/전송 (앱이 포그라운드가 아니어도 동작)
- **user_id 필수 전송**
  - 같은 대표차량을 가진 다른 유저가 동시 로그인 시 데이터 혼탁 방지

---

## 백엔드 (Cloud 서버)

- **무지성 produce** — 수신한 데이터를 검증 후 Kafka로 전송, ClickHouse가 consume하여 적재
- **GPS 튐 기준**: 한반도 영역 안에 들어오는지만 판단
  - 위도: 33.0 ~ 39.0
  - 경도: 124.0 ~ 132.0
  - 범위 밖이면 skip, 로그만 남김

### 검증 메타데이터 (ClickHouse 테이블)

**gps_validation_config** — KEY-VALUE 글로벌 설정

| 컬럼 | 설명 |
|---|---|
| CONFIG_KEY (PK) | 설정 키 |
| CONFIG_VALUE | 설정 값 |
| DESCRIPTION | 설명 |

초기 데이터: `DEFAULT_SPEED_THRESHOLD_KMH = 20.0`

**terminal_zone_config** — 터미널 구역별 속도 임계값

| 컬럼 | 설명 |
|---|---|
| ZONE_ID (PK) | 자동증가 ID |
| ZONE_NAME | 구역명 (신항, 북항 등) |
| CENTER_LAT | 중심 위도 (DECIMAL 10,7) |
| CENTER_LON | 중심 경도 (DECIMAL 10,7) |
| RADIUS_KM | 반경 km (DECIMAL 5,2) |
| SPEED_THRESHOLD_KMH | 속도 임계값 km/h (DECIMAL 5,2) |
| DESCRIPTION | 설명 |

초기 데이터:

| 구분 | 기준점 주소 | 위도 | 경도 | 반경 | 임계값 |
|------|-----------|------|------|------|------|
| 신항 | 경남 창원시 진해구 용원동 1329 | 35.0802408 | 128.8039779 | 5km | 5km/h |
| 신항 배후단지 | 부산 강서구 가락대로 649 | 35.1347164 | 128.8638059 | 3.5km | 5km/h |
| 북항 | 부산 동구 초량동 45-67 | 35.1160086 | 129.0540030 | 3.5km | 5km/h |

### 설정값 관리 방식

- `@PostConstruct`로 서버 기동 시 ClickHouse에서 메모리로 캐싱
- `reload API`로 수동 리로드 (설정 변경 시 호출)
- Redis/Kafka 불필요 — 인스턴스 1대, 설정 변경 빈도 극히 낮음
- Haversine 계산으로 포인트가 터미널 구역 내인지 판단
  - 구역 내 → 해당 `SPEED_THRESHOLD_KMH` 적용
  - 구역 밖 → `DEFAULT_SPEED_THRESHOLD_KMH` 적용

## 장애 대응

| 장애 상황 | 동작 | 앱 처리 |
|---|---|---|
| 프록시 서버 다운 | Bctrans/Allcone backend에서 503 반환 | 앱에서 예외처리 |
| 클라우드 서버 다운 | 프록시가 502/504 반환 → Bctrans/Allcone backend에서 502 반환 | 앱에서 예외처리 |
| Kafka 다운 | Cloud 서버에서 produce 실패 → 502 반환 | 앱에서 예외처리 |
| ClickHouse 다운 | Kafka에 메시지 보존 → ClickHouse 복구 후 자동 소비 | **데이터 유실 없음** |
| Bctrans/Allcone backend 다운 | 앱에서 연결 실패 | 앱에서 예외처리 |

- Kafka 도입으로 ClickHouse 장애 시에도 데이터 버퍼링 가능
- 용도가 밀도 분석(장비 설치 부지 선정)이므로 실시간 트래킹 수준의 신뢰성은 불필요하나, Kafka로 유실 최소화

## 데이터베이스

### 데이터 양 산정

**최악 산정 — 일일 약 244억 건**

```
17만명 동시, 12시간, 30초마다 100개 포인트
(3600 × 12 / 30) × 100 × 170,000 = 24,480,000,000
```

**현실적 산정 — 일일 약 11억 건**

```
2만명 동시 (실 데이터 기반 일일 평균 1~2만명 운송), 9시간, 30초마다 50개 포인트
(3600 × 9 / 30) × 50 × 20,000 = 1,080,000,000
```

### ClickHouse 도입 시

| 항목 | 최악 산정 | 현실적 산정 |
|---|---|---|
| 위경도 정밀도 | 소수점 3자리로 자르면 버팀 | 소수점 4자리로 잘라도 버팀 |
| 압축 효율 | 적은 카디널리티로 압축률 극대화 | 적은 카디널리티로 압축률 극대화 |
| TTL | 90일 충분 | 90일 충분 |
| 컬럼 수 | 14컬럼 전부 가능 | 14컬럼 전부 가능 |
| 서버 구성 | - | 단일 서버 운영 가능 |
| 집계 쿼리 | 빠름 (컬럼 기반 DB 강점) | 빠름 (컬럼 기반 DB 강점) |

### MariaDB 사용 시 (비교 참고)

| 항목 | 최악 산정 | 현실적 산정 |
|---|---|---|
| 집계 쿼리 | 사실상 타임아웃 | 수십 분 ~ 타임아웃 |
| 디스크 I/O | 풀스캔 시 TB 단위 발생 | 파티셔닝 필수 |
| 생존 조건 | 불가 | 아래 조건 **전부** 필요 |

현실적 산정에서 MariaDB가 버티려면:
- 파티셔닝 (일별)
- 인덱스 (LATITUDE, LONGITUDE)
- 집계 결과 별도 테이블로 미리 저장 (배치 집계)
- 원본 삭제 주기 최대한 짧게 (7일 이내)
- 위경도 컬럼만 저장

→ 이 조건을 전부 걸어도 **실시간 집계 불가**

**결론: 이 용도에 맞지 않는 도구**

### ClickHouse vs MariaDB 비교

| 항목 | MariaDB | ClickHouse |
|---|---|---|
| **저장 구조** | 행 단위 → 2컬럼 집계해도 행 전체를 디스크에서 읽음 | 컬럼 단위 파일 분리 → 필요한 컬럼 파일만 읽음 |
| **집계 쿼리 속도** (현실적 산정) | 수십 분 ~ 타임아웃 | 수십 초 이내 |
| **INSERT 처리** | 트랜잭션 오버헤드로 초당 ~2,500건 한계 | 배치 INSERT 기준 초당 수십만 건 |
| **삭제/업데이트** | 빠름 | 느림 (TTL, 파티션 드롭으로 대체) |
| **압축률** | 2~3배 | 5~10배 (반복값 많을수록 극대화) |

→ 수집 데이터 적재 용도상 삭제/업데이트가 없으므로 ClickHouse의 약점이 해당 없음

## Kafka 구성

### 도입 사유

- ClickHouse 장애 시 데이터 버퍼링 (유실 방지)
- 로깅 중앙화 등 향후 다른 서비스 데이터 소스 추가 용이
- 토픽 기반 확장 — 새로운 데이터 파이프라인 추가 시 토픽만 생성하면 끝

### 토픽 설계

| 토픽 | Producer | Consumer | 용도 |
|---|---|---|---|
| `gps-points` | Cloud 서버 (검증 통과 데이터) | ClickHouse (Kafka 엔진 테이블) | GPS 포인트 적재 |
| (향후) `app-logs` | 각 서비스 | ClickHouse | 로깅 중앙화 |

### 데이터 흐름

```
Cloud 서버
  → 검증 통과 → Kafka produce (gps-points 토픽)
  → 검증 실패 → 로그만 남김 (Kafka 전송 안 함)

ClickHouse
  → Kafka 엔진 테이블로 gps-points 토픽 자동 consume → 적재
```

### ClickHouse Kafka 엔진 연동

- ClickHouse는 [Kafka 엔진 테이블](https://clickhouse.com/docs/en/engines/table-engines/integrations/kafka)을 지원
- 별도 consumer 애플리케이션 없이 ClickHouse가 직접 Kafka 토픽을 consume하여 적재
- Materialized View로 Kafka 엔진 테이블 → 실제 저장 테이블로 자동 INSERT

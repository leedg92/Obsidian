# GPS 수집 시스템 아키텍처

## 시스템 구성

```
App → Bctrans/Allcone backend (바이패스) → Proxy (리버스프록시) → Cloud 서버 (검증 + Kafka produce) → Kafka → ClickHouse (consume + 적재)
```

| 컴포넌트        | 위치      | 역할                                 |
| ----------- | ------- | ---------------------------------- |
| App (모바일)   | 클라이언트   | GPS 수집, 필터링(isMoving==0 제외), 배치 전송 |
| Bctrans/Allcone backend | BPA IDC | 바이패스만                              |
| Proxy       | 프록시 서버  | nginx 리버스프록시 설정, 클라우드로 라우팅         |
| Cloud 서버    | 클라우드    | 검증(2단계) + Kafka produce             |
| Kafka       | 클라우드    | 메시지 큐 (토픽: gps-points 등)           |
| ClickHouse  | 클라우드    | Kafka consume + GPS 데이터 적재 + 설정 메타데이터 테이블 |

---

## 앱 (모바일)

- `isMoving == 0`인 포인트는 **제외하고 전송** (정지 상태 데이터 불필요)
- **운송 오더가 존재할 때만** GPS 데이터 수집
  - 단, 백그라운드로 계속 수집/전송 (앱이 포그라운드가 아니어도 동작)
- **user_id 필수 전송**
  - 같은 대표차량을 가진 다른 유저가 동시 로그인 시 데이터 혼탁 방지

---

## 백엔드 (Cloud 서버)

- 수신한 데이터를 **2단계 속도 검증** 후 Kafka produce, ClickHouse가 consume하여 적재
- 검증 실패(skip) 포인트는 Kafka 전송 안 함, 로그만 남김

### 검증 로직 (2단계)

**1단계: 한반도 바운딩 박스 검증**
- `gps_validation_config` 테이블의 좌표 범위로 판단
- 범위 밖이면 즉시 skip (GPS 튐으로 간주)

**2단계: 속도 임계값 검증**
- 포인트 좌표로 Haversine 계산 → 터미널 구역 내인지 판단
  - **구역 내** → 해당 구역의 `SPEED_THRESHOLD_KMH` 적용 (5km/h)
  - **구역 밖** → `DEFAULT_SPEED_THRESHOLD_KMH` 적용 (30km/h)
- 포인트의 speed가 임계값 미만이면 skip

### 검증 메타데이터 (ClickHouse 테이블)

**gps_validation_config** — KEY-VALUE 글로벌 설정

| 컬럼 | 설명 |
|---|---|
| CONFIG_KEY (PK) | 설정 키 |
| CONFIG_VALUE | 설정 값 |
| DESCRIPTION | 설명 |

초기 데이터:

| CONFIG_KEY | CONFIG_VALUE | DESCRIPTION |
|---|---|---|
| DEFAULT_SPEED_THRESHOLD_KMH | 30.0 | 구역 밖 기본 속도 임계값 (km/h) |
| KOREA_MIN_LAT | 33.0 | 한반도 바운딩 박스 최소 위도 |
| KOREA_MAX_LAT | 39.0 | 한반도 바운딩 박스 최대 위도 |
| KOREA_MIN_LON | 124.0 | 한반도 바운딩 박스 최소 경도 |
| KOREA_MAX_LON | 132.0 | 한반도 바운딩 박스 최대 경도 |

**terminal_zone_config** — 터미널 구역별 속도 임계값

| 컬럼 | 설명 |
|---|---|
| ZONE_ID (PK) | 자동증가 ID |
| ZONE_NAME | 구역명 (신항, 북항 등) |
| CENTER_LAT | 중심 위도 (DECIMAL 10,7) |
| CENTER_LON | 중심 경도 (DECIMAL 10,7) |
| RADIUS_KM | 반경 km (DECIMAL 5,2) |
| SPEED_THRESHOLD_KMH | 속도 임계값 km/h (DECIMAL 5,2) |
| DESCRIPTION | 설명 |

초기 데이터:

| 구분 | 기준점 주소 | 위도 | 경도 | 반경 | 임계값 |
|------|-----------|------|------|------|------|
| 신항 | 경남 창원시 진해구 용원동 1329 | 35.0802408 | 128.8039779 | 5km | 5km/h |
| 신항 배후단지 | 부산 강서구 가락대로 649 | 35.1347164 | 128.8638059 | 3.5km | 5km/h |
| 북항 | 부산 동구 초량동 45-67 | 35.1160086 | 129.0540030 | 3.5km | 5km/h |

### 설정값 관리 방식

- `@PostConstruct`로 서버 기동 시 ClickHouse에서 메모리로 캐싱 (두 테이블 모두)
- `reload API`로 수동 리로드 (설정 변경 시 호출)

## 데이터베이스 (ClickHouse)

클라우드 서버 디스크 용량에 따라 설계를 다르게 가져가야함.

### 전략 A: warehouse 단독 (SSD 2TB 이상)

디스크 여유가 충분하면 warehouse에 원본 데이터만 쌓고, 집계 쿼리도 원본에서 직접 수행.

```
Kafka → warehouse 테이블 (전체 12컬럼, TTL 없음)
```

- 5개월 수집 기준 압축 후 **~1.3 TB** → 2TB SSD에서 충분
- mart 불필요 — 집계 쿼리를 warehouse에서 바로 수행 (ClickHouse라 수십 초 이내)

### 전략 B: warehouse + mart (SSD 2TB 미만)

디스크가 부족하면 warehouse에 TTL을 걸고, mart로 밀도 분석용 데이터를 분리 보존.

```
Kafka → warehouse 테이블 (전체 12컬럼, TTL 30일)
          ↓ (Materialized View, INSERT 시 자동)
        mart 테이블 (위경도 분리 4컬럼, TTL 없음 = 영구)
```

- warehouse: TTL 30일, 상시 **~260 GB**
- mart: 영구 보존, 연간 **~400 GB**
- 합산 1년 기준 **~660 GB** → 1TB SSD에서도 운영 가능

### warehouse 테이블 — 원본 데이터

Kafka 엔진 테이블에서 consume한 데이터를 그대로 적재.

| 컬럼            | 타입                | 설명            |
| ------------- | ----------------- | ------------- |
| USER_ID       | String            | 사용자 ID        |
| TRUCK_NO      | String            | 차량번호          |
| TRUCKER_ID    | String            | 운송사 ID        |
| LATITUDE      | Float64           | 위도 (원본 정밀도)   |
| LONGITUDE     | Float64           | 경도 (원본 정밀도)   |
| SPEED         | Float32           | 속도 (km/h)     |
| ACCURACY      | Float32           | GPS 정확도(오차범위) |
| BATTERY_LEVEL | Nullable(Float32) | 배터리 잔량        |
| NETWORK_TYPE  | Nullable(String)  | 네트워크 타입       |
| IS_MOVING     | UInt8             | 이동 여부 (1=이동중) |
| COLLECTED_AT  | DateTime64        | 수집 시각         |
| CREATED_AT    | DateTime64        | 서버 수신 시각      |

- 행당 ~80 bytes (raw), 압축 후 ~8 bytes
- 전략 A: TTL 없음 / 전략 B: TTL 30일

### mart 테이블 — 밀도 분석용 (전략 B에서만 사용)

warehouse INSERT 시 Materialized View로 자동 적재. 위경도를 정수부/소수부로 분리하여 압축 효율 극대화.

| 컬럼      | 타입              | 설명                | 예시 (35.14349, 129.9999) |
| ------- | --------------- | ----------------- | ----------------------- |
| LAT_INT | UInt8 (1byte)   | 위도 정수부            | 35                      |
| LAT_DEC | UInt32 (4bytes) | 위도 소수부 (5자리, ~1m) | 14349                   |
| LON_INT | UInt8 (1byte)   | 경도 정수부            | 129                     |
| LON_DEC | UInt32 (4bytes) | 경도 소수부 (5자리, ~1m) | 9999                    |

- TTL 없음 — 영구 보존
- 행당 10 bytes, 정수부(LAT_INT, LON_INT)는 부산항 일대 거의 상수 → 사실상 0 바이트로 압축
- 집계 시 원본 좌표 복원: `LAT_INT + LAT_DEC / 100000.0`
- 10m 그리드 집계: `LAT_DEC / 10`으로 4자리 단위 GROUP BY

### 예상 디스크 사용량 (현실적 산정 기준)

```
일일 약 10.8억 건 (2만명 동시, 9시간, 30초마다 50개 포인트)
```

| 전략                           | 5개월     | 1년      | 필요 디스크         |
| ---------------------------- | ------- | ------- | -------------- |
| A (warehouse 단독)             | ~1.3 TB | ~2.6 TB | **SSD 2TB 이상** |
| B (warehouse TTL 30일 + mart) | ~460 GB | ~660 GB | **SSD 1TB 이상** |
| B (warehouse TTL 30일 + mart) | ~460 GB | ~660 GB | **SSD 1TB 이상** |

## 참고: DB 선정 근거

### 데이터 양 산정

**최악 산정 — 일일 약 244억 건**

```
17만명 동시, 12시간, 30초마다 100개 포인트
(3600 × 12 / 30) × 100 × 170,000 = 24,480,000,000
```

**현실적 산정 — 일일 약 11억 건**

```
2만명 동시 (실 데이터 기반 일일 평균 1~2만명 운송), 9시간, 30초마다 50개 포인트
(3600 × 9 / 30) × 50 × 20,000 = 1,080,000,000
```

### ClickHouse 도입 시

| 항목 | 최악 산정 | 현실적 산정 |
|---|---|---|
| 위경도 정밀도 | 소수점 3자리로 자르면 버팀 | 소수점 4자리로 잘라도 버팀 |
| 압축 효율 | 적은 카디널리티로 압축률 극대화 | 적은 카디널리티로 압축률 극대화 |
| TTL | 90일 충분 | 90일 충분 |
| 컬럼 수 | 14컬럼 전부 가능 | 14컬럼 전부 가능 |
| 서버 구성 | - | 단일 서버 운영 가능 |
| 집계 쿼리 | 빠름 (컬럼 기반 DB 강점) | 빠름 (컬럼 기반 DB 강점) |

### MariaDB 사용 시 (비교 참고)

| 항목      | 최악 산정          | 현실적 산정          |
| ------- | -------------- | --------------- |
| 집계 쿼리   | 무조건 타임아웃       | 수십 분 ~ 타임아웃     |
| 디스크 I/O | 풀스캔 시 TB 단위 발생 | 파티셔닝 필수         |
| 생존 조건   | 불가             | 아래 조건 **전부** 필요 |

현실적 산정에서 MariaDB가 버티려면:
- 파티셔닝 (일별)
- 인덱스 (LATITUDE, LONGITUDE)
- 집계 결과 별도 테이블로 미리 저장 (배치 집계)
- 원본 삭제 주기 최대한 짧게 (7일 이내)
- 위경도 컬럼만 저장

→ 이 조건을 전부 걸어도 **실시간 집계 불가**

**결론: 이 용도에 맞지 않는 도구**

### ClickHouse vs MariaDB 비교

| 항목                    | MariaDB                        | ClickHouse                  |
| --------------------- | ------------------------------ | --------------------------- |
| **저장 구조**             | 행 단위 → 2컬럼 집계해도 행 전체를 디스크에서 읽음 | 컬럼 단위 파일 분리 → 필요한 컬럼 파일만 읽음 |
| **집계 쿼리 속도** (현실적 산정) | 수십 분 ~ 타임아웃                    | 수십 초 이내                     |
| **INSERT 처리**         | 트랜잭션 오버헤드로 초당 ~2,500건 한계       | 배치 INSERT 기준 초당 수십만 건       |
| **삭제/업데이트**           | 빠름                             | 느림 (TTL, 파티션 드롭으로 대체)       |
| **압축률**               | 2~3배                           | 5~10배 (반복값 많을수록 극대화)        |

→ 수집 데이터 적재 용도상 삭제/업데이트가 없으므로 ClickHouse의 약점이 해당 없음

## 도메인 지식 요약 (매 세션 자동 로드)

### 시스템 구성

- 우리 회사 : smartm2m

| 시스템       | 역할                                  | 관리 주체                | 위치     |
| --------- | ----------------------------------- | -------------------- | ------ |
| BaaS      | 블록체인 플랫폼                            | smartm2m             | 터미널 외부 |
| TOS       | Terminal Operating System           | 각 터미널                | 터미널 내부 |
| TA        | Terminal Agent (TOS↔BaaS 중계)        | smartm2m             | 터미널 내부 |
| bctrans   | Backend API (TSS오픈API + Allcone모바일) | 컨테인어스→smartm2m 인수 예정 | 터미널 외부 |
| Allcone 앱 | 모바일 클라이언트                           | 컨테인어스→smartm2m 인수 예정 | 클라이언트  |

### 망 구분

- **EDI망 (VBS)**: 운송사 ↔ 중계사(KTNET) ↔ 터미널. 고유키 = `PIN_NO`
- **블록체인 망 (TSS)**: 운송사 ↔ BaaS ↔ TA. 고유키 = `DOC_KEY`
- 두 망은 경유 인프라만 다르고 역할은 동일 (운송 데이터 중계)
- 고유키는 항상 BaaS가 생성

### 터미널 운송건 관점 (핵심 도메인 룰)

- 터미널 입장에서는 **모든 운송건이 단일건**이다.
- 관심사: "이 차량이 컨테이너를 싣거나/내린다"
- 비관심사: "이 차량이 다음에 어디로 갈지" (VBS/TSS 구분, 환적 여부)
- 따라서 TSS(환적)도 터미널 레벨에선 두 개의 단방향 트랜잭션으로 분해되며, 그룹오더도 결국 개별 차량 단위로 처리된다.
- 이 룰을 알면 "왜 어떤 인터페이스는 VBS/TSS 구분이 있고 어떤 건 없는지" 자연스럽게 설명된다.

### 핵심 용어

| 용어            | 의미                            |
| ------------- | ----------------------------- |
| COPINO        | 운송사가 요청한 운송건 = 운송오더           |
| e-Slip (인수도증) | 터미널 발급, 게이트 진입 시 작업 지시서       |
| VBS           | 운송예약서비스 + EDI망 운송건 통칭         |
| TSS (= ITT)   | 환적 운송. 반출→반입 두 터미널을 묶은 논리적 개념 |
| 그룹오더          | TSS를 여러 차량 × 여러 컨테이너로 묶은 단위   |
| 냉컨            | 냉동 컨테이너 (냉동기라고 하지 않음)         |
| ECO            | Exception Cargo Operation — 예외 컨테이너 처리소 (터미널 단일 운송건 관점으로 처리, `REQ_GATE_IN_ECO`) |
| 내부통행          | PNIT-PNC 인접 터미널 사이의 직접 통행 문 — TSS/복화 케이스에서 사용 (`REQ_INTERNAL_GATE_IN`) |
| 복화             | 컨테이너 하차 직후 같은 차량에 다른 컨테이너를 실어서 나오는 흐름 — 앱 "복화 버튼" 활성화 (상세 미정리) |

### 무브먼트 공통 패턴 (VBS/TSS 공통)

오더 생성 이후: TOS 이벤트 → TA → BaaS → bctrans invoke alarm → 앱 푸시

### bctrans 진입점

| 망 | URL | Controller |
|----|-----|------------|
| VBS | POST /vbs/invoke/alarm | VbsInvokeAlarmController |
| TSS | POST /itt/invoke/alarm | IttInvokeAlarmController |
| 그룹오더 | POST /tss/group/* | TssGroupOrderController |

### 프로세스 흐름 한줄 요약

| 흐름         | 요약                                                         | 상세 문서                                       |
| ---------- | ---------------------------------------------------------- | ------------------------------------------- |
| VBS 오더     | 운송사→KTNET(or KLNET)→터미널, 검증결과 수신→오더 확정                     | `docs/claude-ref/vbs-flow/`                 |
| TSS 오더     | 운송사→BaaS(DOC_KEY)→TA·bctrans에 CreateCopino alarm→GetCopino 조회→TA는 MIG/TOS, bctrans는 DB/앱 | `docs/claude-ref/tss-flow/`                 |
| 그룹오더       | 운송사→bctrans→BaaS(DOC_KEY)→TA(StartGroupOrder)→진입 시 컨테이너 확정 | `docs/claude-ref/tss-flow/GroupOrder.md`    |
| VBS 터미널이벤트 | GateIn/GateOut/JobDone → 상태 갱신 + 앱 알림                      | `docs/claude-ref/vbs-flow/TerminalEvent.md` |
| TSS 터미널이벤트 | 위와 동일 패턴 + nextJob 처리                                      | `docs/claude-ref/tss-flow/TerminalEvent.md` |
| 자동화 블록     | 차량 자동화 블록 진입 → TOS→TA→BaaS→bctrans→앱 알림 → 기사 "준비완료" → 역방향 TOS 통지 → 자동 크레인 상/하차 (양방향) | (사용자 답변 기반, 미문서화) |

### TSS 이벤트 분류 (IttInvokeAlarmController)

| 분류 | method |
|------|--------|
| 그룹오더 시작/재시작 | StartGroupOrder, RestartGroupOrder |

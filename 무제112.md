# BNCT 지오펜스 통보 수정 회고 (BPABCTRANS-251)

> 지오펜스 진입 이벤트를 BNCT로 통보하는 기능에서 발견된 문제들과, 그걸 어떻게 분석하고 고쳤는지에 대한 기록.
> 결과물뿐 아니라 **중간에 방향이 바뀐 지점**을 남겨서 다음에 비슷한 판단을 할 때 참고할 수 있게 정리한다.

---

## 1. 어떤 문제가 있었나

지오펜스 감지 → BNCT 데이터 전송 자체는 동작했지만 두 가지 이슈가 있었다.

### 이슈 A — 같은 차량 통보가 BNCT로 중복 전송
앱은 한 번 영역을 통과할 때 **BNCT 운송오더 개수만큼** `areaEvent`를 반복 호출한다(오더가 2개면 2번, 3개면 3번).
각 호출은 `transportCode`만 다르고 truck/area/event는 같다. 백엔드는 매 호출마다 `collect(truckNumber)`로
차량 전체 오더를 모아 통보하므로, **같은 payload가 오더 수만큼 BNCT로 중복 전송**됐다.

### 이슈 B — 이미 끝난 운송오더가 통보 리스트에 섞임
통보 대상 오더를 만들 때 완료 판정이 느슨했다. "게이트아웃 안 된 건"이 기준이었는데 터미널이 게이트아웃
이벤트를 누락하는 경우가 있어, **이미 작업이 끝난 오더가 계속 리스트에 남는** 문제가 있었다.
실제 사례: 오전 11시에 잡던(JOB_DONE) 끝난 "미지정공컨 반출" 오더가, 오후 2시 47분 진입 통보에 그대로 실려 나옴.

---

## 2. 이슈 A — 중복 전송: 세 번 방향을 바꿨다

핵심은 "무엇이 동일한가"였다. 기존 `idempotencyKey`는 `transportCode`를 포함해서 **오더 단위**라 이 케이스를
못 막는다. 통보는 오더와 무관하게 `(truckNumber + areaId + eventTp)`로 결정되므로 그 기준으로 중복을 잡아야 했다.

| 시도 | 방식 | 왜 접었나 |
|---|---|---|
| 1차 | **신규 dedup 테이블 + SHA-256 멱등키 + 시간버킷 원자적 INSERT** | 과설계. 새 테이블·해시·버킷까지 도입할 문제가 아니었음 |
| 2차 | **인메모리 Guava 캐시(5초 만료)** | 여전히 새 상태(캐시) 추가. 부하분산 환경 보장도 애매 |
| **최종** | **기존 `allcone_user_area_event` 조회로 판정 (락 없음)** | 이미 커밋되는 데이터를 그대로 활용 — 가장 단순 |

### 최종 로직
`recordAreaEvent`가 각 이벤트를 **동기 커밋**한 뒤 비동기 통보가 도는 구조를 이용한다. 전송 직전에
`(truck+area+event)`로 최근 5초 내 **최초 삽입 이벤트(EVENT_ID 최소)의 transportCode**를 조회하고,
그게 내 transportCode일 때만 전송한다.

```sql
SELECT TRANSPORT_CODE FROM allcone_user_area_event
WHERE TRUCK_NUMBER=? AND AREA_ID=? AND EVENT_TP=?
  AND EVENT_DTM >= NOW(3) - INTERVAL 5 SECOND
ORDER BY EVENT_ID ASC LIMIT 1
```

**락은 왜 안 썼나** — N개 스레드가 각자 조회해도 "최초 EVENT_ID" 승자는 **결정론적으로 하나**라 조율이 필요 없다.
- 비관락(FOR UPDATE / GET_LOCK): DB 경합만 늘고 실익 없음.
- 낙관락(버전컬럼): read-only 판정이라 애초에 대상 아님.
- 남는 리스크는 커밋 가시성 타이밍 엣지뿐이고, 그때도 "몇 건 중복" 정도로 graceful degrade → 기존보다 무조건 나음.

추가 이점: `transportCode`는 스레드가 이미 request로 들고 있어 **record→notify로 넘길 값이 전혀 없음** → 컨트롤러/DTO 무변경.

## 3. 이슈 B — 완료 오더 제외: 기준을 두 번 바꿨다

처음엔 "게이트인 전만 남긴다"로 시작했는데, 실제 데이터/앱 동작을 확인하며 기준이 정교해졌다.

### 3-1. VBS: 게이트인 → 앱과 동일 규칙(jobDone + gateOut)
- 처음: `GATE_IN_DTM == null`(게이트인 전)만 통보.
- 확인해보니 앱의 운송오더 조회(`/mobile/trans/order/list`, `VbsTransOrderService:521`)는
  **`JOB_DONE_DTM == null && GATE_OUT_DTM == null`** 을 노출 기준으로 쓴다.
- 결정적 근거: **"미지정공컨 반출 → 재할당"** 케이스에서 게이트인 기록이 원래 미지정 오더에 붙고 재할당된 오더엔
  `GATE_IN_DTM`이 비어 있을 수 있다. 그러면 게이트인 기준으론 **완료건이 다시 노출**되지만, `JOB_DONE`은 확실히 잡는다.
- → **앱과 동일 규칙으로 정렬**. 게이트아웃이 누락돼도 JOB_DONE이 백업으로 완료건을 차단하므로 원래 걱정하던
  "게이트아웃 이벤트 누락" 문제도 함께 해결.

### 3-2. TSS: leg별 게이트 구간 + 취소 제외
- 반출/반입은 한 오더가 두 터미널에 기여하므로 leg별로 나눠 판정한다.
- 반입 leg 조건도 한 번 바뀜: 처음엔 `반출 게이트아웃 이후`(FROM_GATE_OUT)로 봤다가,
  게이트아웃 불안정성을 감안해 **게이트인 기준**으로 정리:
  - 반출(OUT): `FROM_GATE_IN_DT == null` (반출터미널 게이트인 전)
  - 반입(IN): `FROM_GATE_IN_DT != null && TO_GATE_IN_DT == null` (반출 게이트인 후 ~ 반입 게이트인 전)
- 앱은 TSS를 게이트시각이 아니라 `transStatus`(1~100 & 미취소)로 노출하며 **반출 게이트아웃 후에도 COMPLETE 전까지 유지**한다.
  IN-leg 조건이 이 구간을 그대로 커버함을 확인.
- 앱 대비 유일한 갭이던 **취소 제외**(`OUT_CANCEL_TIME`/`IN_CANCEL_TIME`)를 추가.

## 4. 부수 작업 — 전송 로그 (디버깅용)
BNCT로 실제 보낸 요청 payload와 결과를 매 전송마다 `allcone_geofence_notify_log`에 적재.
(요청 JSON·URL·성공여부·메시지·시도횟수. 성공/실패 모두 기록, 적재 실패는 통보 흐름에 영향 없음.)
→ 이건 기존 테이블에 없는 아웃바운드 집계 payload라 신규 테이블이 맞다(중복제거 때와 달리 재사용 대상 없음).

## 5. 이번 작업에서 얻은 판단 기준

- **중복/멱등 문제는 "새 자산을 늘리기 전에, 이미 커밋되는 데이터로 결정론적으로 풀 수 있나?"를 먼저 본다.**
  이번엔 그게 됐고(최초 EVENT_ID), 그래서 테이블·캐시·락 전부 불필요했다.
- **"완료" 판정은 도메인의 정본(앱 조회 규칙)에 맞춘다.** 직관("게이트인 전")보다, 실제 앱이 뭘 숨기는지가 정답이었다.
  게이트 이벤트가 불안정할 때 JOB_DONE 같은 상위 신호를 백업으로 두면 더 견고하다.
- **VBS와 TSS는 완료/노출 판정 축이 다르다.** VBS는 jobDone/gateOut, TSS는 transStatus 기반. 하나로 뭉뚱그리면 틀린다.

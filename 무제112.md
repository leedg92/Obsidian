# bctrans → allcone 커플링 분석 (1차)

> 작성일: 2026-04-15
> 브랜치: `feature/bctransApiForAllcone/WIP`
> 목적: 신 allcone이 호출할 bctrans API 목록 도출 전 단계 분석

## 파일별 bctrans 의존 요약 (실 API화 대상)

### 1. `AllconeTransOrderDomainService` ⚠️ 특수 케이스
- 의존: `ContainerTransportStatusDomainService`, `ContainerTransportSummaryDomainService`, `ContainerTransportSummaryDomainRepository`, `ContainerTransportApplicationRepository`
- 핵심 로직: `filteringVbsTransportOrderDataForPinNoDuplicate` — 핀번호 중복 시 인공 GateIn 이벤트 생성 + DB 갱신 (트랜잭션)
- **판단**: 단순 wrapper로 API화 부적절. 이 로직 **자체를 bctrans로 이동**하는 게 맞음 (bctrans 데이터를 만지는 로직이므로)
- 신 allcone이 호출할 API: `POST /bctrans/vbs/transport-order/filter-pin-duplicate` (리스트 in/out)

### 2. `AllconeVbsTruckTransOrderService`
- 의존: `ContainerTransportVbsDomainService`, `ContainerTransportDomainRepository`
- 주요 bctrans 호출:
  1. `containerTransportVbsDomainService.neededUpdateVbsForByConfirmAppointment(orderId, reservationTime)` → 읽기
  2. `containerTransportVbsDomainService.updateContainerVbsByPinNo(needUpdateVbs, pinNo)` → 쓰기
  3. `containerTransportDomainRepository.getContainerTransportByPinNo(pinNo)` → 읽기
- 참고: 이미 **webClientService로 bctrans HTTP 호출**하는 메서드들도 존재 (`updateTruckTransStatusOnRoadRequestToBctrans`, `cancelStartToBctrans`, `requestBlockInToBctrans`, `requestDigitalGatein`)
- 필요 API:
  - `GET /bctrans/vbs/container-transport?pinNo=...` → 읽기
  - `GET /bctrans/vbs/container-transport-vbs/for-confirm-appointment?orderId=&reservationTime=` → 읽기
  - `POST /bctrans/vbs/container-transport-vbs/update-by-pin` → 쓰기

### 3. `AllconeVbsTruckTransOrderRepository` ⚠️ 이번 범위 **제외** (사용자 결정)
- 의존: `AllconeContainerTransportOrderMapper`, `ContainerTransportMapper`, `ContainerTransportVbsMapper` (bctrans 소유)
- 사용 메서드(호출처에서 본 것):
  - `getAllconeTransportOrderDataByPinNo` (다수 호출)
  - `getContainerTransportByPinNo`
  - `updateCancelAppointment`, `updateSingleAppointment`, `updateRequestAppointment`
  - `updateTruckTransStatusOnRoad`, `updateTruckTransOrderLatestStatusInZone`
  - `getContainerTransportVbsByTransportCode`
- **사용자 결정**: Mapper 직접 참조는 이번 Phase에서 **그대로 유지** (내부 함수 호출). 신 allcone 분리 시 별도 라운드에서 처리
- **판단 유지 (참고용)**: Repository가 타 도메인 Mapper를 직접 주입받는 건 심각한 레이어 위반이므로, 추후 라운드에서는 bctrans 측 Service 신설 후 API화 권장

### 4. `IttTruckTransStatusUpdateService` (TSS 측 커플링)
- 의존: `TssGroupOrderProcessService`, `BctransIttTruckTransOrderService`, `PushTruckerService`
- 주요 호출:
  1. `tssGroupOrderProcessService.saveGroupOrderTransStatus(param)` → 쓰기
  2. `pushTruckerService.send(param)` → FCM
  3. `bctransIttTruckTransOrderService.updateTssOrderTransStatusMetrics(param)` → 쓰기
- 참고: 이미 `/mobile/trans/itt/status`, `/mobile/trans/itt/cancelStart`, `/mobile/trans/itt/req/blockin` HTTP 호출 메서드 존재
- 필요 API:
  - `POST /bctrans/tss/group-order/save-trans-status`
  - `POST /bctrans/tss/push-trucker`
  - `POST /bctrans/tss/itt-trans-order/update-status-metrics`

### 5. DTO/Util 파일들 (enum만 참조)
- `VbsOrderData`, `TssTransOrderData`, `CommonOrderData`, `CommonInternalTransOrderData`, `TransportFrameData`, `TransOrderDataItem`, `EslipDataItem`, `CanIssueSlipData`, `OrderToDataItemUtil`, `ContainerHandlerRequestDto`, `TerminalContainerSwapRequestDto`, `TerminalContainerChangeDirectionRequestDto`, `InOutTypeDeserializer`
- **처리**: enum 복사 외 구조 변경 불필요

### 6. 기타 Entity/VO 참조
- `UserDigitalGateInRequestEntity` — `ContainerTransport` 사용 (fromEDIData 팩터리 메서드 인자)
- `VbsTruckTransOrderController` — `BctransVbsTruckTransOrderService` 직접 주입
  - **주의**: 이 Controller는 `allcone/vbs/web/`에 있지만 **공유 컨트롤러**. `serverName` 프로필에 따라 allcone 서비스(HTTP 경유) 또는 bctrans 서비스(내부 직접)를 분기 호출
  - 즉 "allcone → bctrans 내부 함수 호출"이 아니라 **bctrans WAR 내부에서 자기 서비스 호출**이므로 **API화 대상 아님**
- `ContainerHandlerRequestDto` — `AllconeTransportOrderData`, `TssTransportOrderData` VO 사용

→ **VO는 allcone 쪽에 복사**하고 DTO로 취급. Entity 의존은 API 응답 DTO로 대체.

## 이미 존재하는 HTTP 호출 패턴 (참고용)

allcone 코드 내 일부 메서드는 이미 `webClientService.sendRequest(bctransUrl + "/mobile/trans/...")` 형태로 **HTTP로 bctrans를 호출**하고 있음:

| 엔드포인트 | 호출처 |
|---|---|
| `POST /mobile/trans/vbs/status` | `updateTruckTransStatusOnRoadRequestToBctrans` |
| `POST /mobile/trans/vbs/cancelStart` | `cancelStartToBctrans` |
| `POST /mobile/trans/vbs/req/blockin` | `requestBlockInToBctrans` |
| `POST /mobile/trans/vbs/req/digitalGatein` | `requestDigitalGatein` |
| `POST /mobile/trans/itt/status` | `updateTransStatusRequestBctrans` |
| `POST /mobile/trans/itt/cancelStart` | `cancelStartToBctrans` (ITT) |
| `POST /mobile/trans/itt/req/blockin` | `requestBlockInToBctrans` (ITT) |

→ 신규 API는 이 네이밍 관례를 따르는 게 자연스러움 (`/mobile/trans/...` 하위 또는 `/internal/...` 하위). **실 경로는 논의 필요**.

## 전환 방식 분류

| 타입 | 건수 | 전략 |
|---|---|---|
| A. enum/상수 복사 | 40 | 단순 복사 |
| B. Service 메서드 wrapper API | **7** (VBS 4 + TSS 3) | Controller 생성, 얇은 wrapper — #1, #2, #4, #5, #8, #9, #10 |
| C. Repository 직접 참조 → Service → API | **3** | bctrans 측에 Service 선추가 후 Controller wrapper — #3, #6, #7 |
| D. 로직 이동 (allcone 로직을 bctrans로) | 1 흐름 (`filteringVbsTransportOrderDataForPinNoDuplicate`, #4~#7 묶음) | API 호출로 바꾸고 로직을 bctrans로 이관 |
| E. Entity/VO를 DTO로 변환 | ~11 | 응답 DTO 별도 설계 |
| F. Mapper 직접 참조 | 11 | **이번 범위 제외** (사용자 결정, 내부 함수 유지) |

**합계 B+C = 10개 메서드** (로직 단위 묶으면 7~8 엔드포인트)

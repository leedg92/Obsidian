FCM 알림이 계속 수신되는 원인 분석해보았으며, 우선 말씀하신 부분의 약간 모호한 부분과 추가적으로 짐작되는 부분들 공유 드리겠습니다.

- **같은 기기 + 같은 대표차량을 사용할 다른 사용자 로그인 (토큰 관점)**
    - 데이터베이스 관점에서 토큰은 유니크키이기에 중복이 존재할 수 없는 구조입니다.   
        그래도 발생가능한 원인을 꼽자면 "기존 아이디/토큰삭제 실패 ->  새 아이디/토큰 인서트 실패(디비레벨 에러)"입니다.  
        
- **같은 기기 + 같은 대표차량을 사용할 다른 사용자 로그인 (대표차량 관점)** 
    - 소스코드상 같은 대표차량의 중복 ACT가 발생가능하긴 합니다.
        - 정상흐름은 교대한 B가 "**대표차량 설정 -> A 동일 대표차량 해지 -> A 해지 FCM -> B 대표차량 update**"으로 진행됩니다.
        - 모든 로직이 한 함수내에서 개별 트랜잭션으로 진행됩니다.
        - 이걸 미루어보아  "**A 해지 FCM -> B 대표차량 update"** 사이에 A가 다시 앱을 켜 대표차량을 다시 뺏어오려고 할때 ACT인 대표차량이 두개 존재할 가능성이 있습니다 .
            - 이건 "**다른 기기 + 같은 대표차량을 사용할 다른 사용자 로그인**" 시의 대표차량 관점에서도 동일하게 작용합니다.
        - 다만 이는 그럴 이유가 없을 뿐더러, 실제로 할려고해도 "FCM을 받고 DB에 대표차량이 B소유로 변경되는" 그 밀리초사이에 A가 대표차량을 뺏어오는게 사실상 불가능합니다. 디비조회상 대표차량 ACT인 로우가 2건이상 잡힌게 하나도 없구요.

다만 위의 두 경우에는 애초에 "같은기기"의 전제가 걸리게 됩니다.

같은 기기라는 말은 "회사에서 제공하는 공기계"일 경우라는 뜻인데,   
이는 공기계를 가지고 퇴근해 집에서 알림을 받았단 뜻이며 이후 교대한 인원은 핸드폰 없이 운송을 하고 있단 의미가 됩니다.

고로 해당 시나리오를 제외하고

예상되는 시나리오는 "**다른 기기 + 같은 아이디**"입니다.

- 정상 시나리오 : A 퇴근 -> B 로그인 -> A 강제 로그아웃 -> A FCM 미수신 / B FCM 수신
- 비정상 시나리오는 기본적으로 두 기기가 로그인 상태를 서로 가질려고 할때 문제가 생깁니다.
    - A 퇴근 -> B 로그인 -> A 강제 로그아웃 -> A 로그인(강제 로그아웃 인지 혹은 습관적으로) -> B 강제 로그아웃
    - A 퇴근 -> B 로그인안함(==앱 안켬) -> A 로그인 상태
        - 이 경우 마지막 로그인한 계정에 FCM 알림이 오게 됨으로 A기사가 알림을 받게 됩니다.
        - A가 FCM 알림을 받은 이후 B가 운송오더 확인 혹은 GateIn하여 인수도증을 확인하고 싶을때 다시 로그인(앱 킴)하게되면 A가 강제 로그아웃되면서 이후 정상적으로 B가 알림을 받게 됩니다.
- 결론적으로 "마지막으로 로그인한 기기가 B의 기기가 아닌 경우" 해당 상황이 발생하게 됩니다

만약 이게 원인이면 이에 대한 해결방안으로는 같은 아이디로 로그인 시 "접속중인 기기가 있다. 로그아웃시키겠냐"라는 맥락의 팝업을 띄워주는게 일단 현재로서는 최선일듯합니다.

운영 올컨e 로그를 통해 강제로그아웃 트리거가 얼마나 발생했는지 확인하면 가장 파악이 빠르지만 제 자리에서는 현재로서는 bctrans의 로그만 조회가능하기에 추가적으로 allcone로그 확인이 필요합니다.

우선 비슷한 민원이 다시 발생할 시 해당 차량으로 allcone로그를 그랩해 확인해 원인을 좁히는 방향이 좋아보입니다.

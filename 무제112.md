# 패키지 간 침범 전수 조사

> 작성일: 2026-04-20
> 목적: MSA 전환 계획(`msa-architecture-draft.md`, `msa-migration-plan.md`) 기준으로 현재 `api-bctrans` 코드베이스의 패키지 경계 위반을 정방향/역방향 양쪽 전수 조사

## 패키지 소속 기준 (전환 계획 기준)

| 소속 | 패키지 |
|------|--------|
| **allcone 소속** | `allcone/`, `gps/`, `insa/` |
| **bctrans 소속** | `bctrans/`, `invoke/` |
| 양쪽 복사 | `healthCheck/` |
| 분배 대기 | `common/` |
| 폐기 검토 | `oldbctrans/`, `dataLinkage/` |

## 침범 유형 정의

| 유형 | 조건 |
|------|------|
| **침범 A (이미 HTTP)** | `${bctrans.url}` / `${allcone.url}` 등 HTTP 래핑 호출. 분리 준비된 상태 |
| **침범 B (DB 직통)** | Mapper/Repository가 타 소속 테이블 직접 접근 |
| **침범 C (내부 함수)** | `import`로 타 소속 클래스 주입/호출 |

테이블 소유:
- `allcone_*`, `user_regist_truck`, `user_digital_gate_request` → allcone
- `general_*`, `terminal_*`, `container_*`, `tb_b_*`, `bctrans_*`, `tss_*` → bctrans
- `chainportaldb.*` → 외부 DB (침범 아님)

---

## 1. 요약

| 방향 | 침범 A | 침범 B | 침범 C |
|------|--------|--------|--------|
| **정방향 (allcone → bctrans)** | 17건 | 10개 mapper, 59건 | 22개 파일, 65건 |
| **역방향 (bctrans → allcone)** | 0건 | 0건 | 12건 |

### common 편향
- allcone 쪽만 참조: ~71 파일
- bctrans 쪽만 참조: ~82 파일 (`tss/`, `vbs/` 핵심)
- 양쪽 참조: ~30 파일 (`response`, `support`, `config`, `util`)

### 레거시 패키지
- `dataLinkage`: 자가완결, API 미호출 → 폐기 후보
- `oldbctrans`: 자가완결, 고립 → 폐기 후보
- `healthCheck`: 양쪽 URL HTTP 호출 사용 중 → 양쪽 복사 계획 유지
- `gps`, `insa`: Clean (`common.response`/`support`만 참조) → allcone 소속 확정 안전

## 2. 정방향 침범 (allcone → bctrans)

### 2-1. 침범 A — 17건 (이미 HTTP 전환 완료)

`allcone-api-mapping-verification.md` 3.1절과 일치. 신규 작업 불필요.

### 2-2. 침범 B — DB 직통, 10개 mapper 파일

| Mapper 파일 | 대상 테이블 | 엔드포인트 |
|-----------|-----------|----------|
| `VbsTransOrderMapper.xml` | `container_transport*` (6) | `/mobile/trans/order/list` |
| `TssTransOrderMapper.xml` | `tb_b_truck_trans_odr*` (3) | `/mobile/trans/order/list` 내부 |
| `EslipDataMapper.xml` | `container_transport*` (8+) | `/mobile/trans/order/eslip` |
| `GeofenceMapper.xml` | `terminal_area` | `/mobile/geofence/digitalGateArea` |
| `GateInWithOutCopinoMapper.xml` | `terminal_truck_event` | 내부 로직 |
| `ContainerHandlerReasonMapper.xml` | `bctrans_container_handler_reason` | 내부 로직 |
| `NoticeMapper.xml` | `general_terminal` (단순 라벨) | `/mobile/notice/*` |
| `TtsMessageMapper.xml` | `tb_b_tss_group_order_m` | 내부 로직 |
| `AlarmMapper.xml` | `allcone_*` (침범 아님) | — |
| `AlarmConfigMapper.xml` | `allcone_*` (침범 아님) | — |

**6개는 이미 `AllconeBridgeController`로 커버됨. 신규 필요 6개는 `allcone-api-mapping-verification.md` 3.2절 참조.**

### 2-3. 침범 C — 내부 함수 호출, 22개 파일, 65건

| 파일 | 주입 대상 | 유형 |
|------|---------|------|
| `AllconeVbsTruckTransOrderService` | `ContainerTransportVbsDomainService`, `ContainerTransportDomainRepository` 외 | B+C 복합 |
| `AllconeVbsTruckTransOrderRepository` | `ContainerTransportMapper`, `ContainerTransportVbsMapper`, `AllconeContainerTransportOrderMapper` | B+C 복합 |
| `AllconeTransOrderDomainService` | `ContainerTransport{Status,Summary,Application}DomainRepository`, 2개 Service | B+C 복합 |
| `VbsTruckTransOrderController` | `BctransVbsTruckTransOrderService` | C |
| `IttTruckTransStatusUpdateService` | `TssGroupOrderProcessService`, `BctransIttTruckTransOrderService`, `PushTruckerService` | C (HTTP 전환 후에도 필요) |
| allcone VO/DTO 다수 | `InOutType`, `FeType`, `CommodityType` 등 `bctrans.common.enums` | enum 공유 40+ |

**핵심: `bctrans.common.enums`가 잘못된 위치. allcone 18파일 + bctrans 67파일 + common 10파일 모두 참조 → enum을 common으로 이동하거나 양쪽 복사 필수.**

## 3. 역방향 침범 (bctrans → allcone)

### 3-1. 침범 A, B — 0건

bctrans에서 `allcone.url` HTTP 호출, allcone 소유 테이블 직접 접근 모두 미발견.

### 3-2. 침범 C — 12건

| 파일 | 주입 대상 | 용도 |
|------|---------|------|
| `AllconeBridgeController` | `AllconeTransOrderDomainService`, `VbsTransOrderService`, `AllconeVbsTruckTransOrderRepository` | bridge API 구현 |
| `BctransVbsTruckTransOrderService` | `AllconeVbsTruckTransOrderService`, `BctransDigitalGateinRequest` | 공유 Service 호출 |
| `ContainerTransportStatusDomainService` | `VbsOrderData` (VO) | VO 참조 |
| `BctransIttTruckTransOrderService` | `TransshipmentTransStatus` (VO) | VO 참조 |
| `invoke/` 패키지 | `Sign*Service`, `Alarm*Service`, `AllconeVbs*Service` | allcone 알림/FCM 호출 (handler 역할) |

**특징: 단방향 프록시 패턴 또는 VO 공유. 비교적 약한 결합. 기존 문서 "5건" → bridge 구현으로 현재 12건.**

## 5. 레거시/폐기 검토

### 5-1. dataLinkage (6파일)
- 자가완결, API 미호출
- 용도: `TransportRecodeMapper` 운송 동기화 (비활성)
- **판정**: Phase 1에서 재확인 후 폐기 가능

### 5-2. oldbctrans (14파일)
- 자가완결, 외부 참조 0건 (고립)
- 용도: 레거시 VBS/ITT/Eslip 로직 (Old*Service)
- **판정**: Controller 미등록 확인 후 폐기 가능

### 5-3. healthCheck
- `allcone.url`, `bctrans.url` 양쪽 HTTP 호출 사용 중
- **판정**: 양쪽 복사 계획 유지

### 5-4. gps (5파일)
- `common.response`, `util`만 참조 → Clean
- **판정**: allcone 소속 확정 안전

### 5-5. insa
- `common.support` + allcone `DeviceAuthService` 협력 → Clean
- **판정**: allcone 소속 확정 안전

## 6. 기존 계획 대비 현황

| 항목 | 계획 문서 | 실제 조사 |
|------|---------|---------|

| gps/insa 오염 여부 | "검증 필요" | Clean |
| dataLinkage, oldbctrans 폐기 | "확인 후 결정" | 둘 다 고립, **폐기 가능** |

## 7. 리스크 / 후속 검토 필요 항목

### 7-1. 트랜잭션 경계 문제
allcone이 bctrans의 Service/Repository를 직접 조작하며 **하나의 트랜잭션**으로 동작 중. HTTP 분리 시 Two-Phase Commit 또는 Saga 패턴 필요.

대표 사례:
- `AllconeTransOrderDomainService.gateInTimeCorrectionDb()` — bctrans Repository 직접 UPDATE
- `AllconeVbsTruckTransOrderService.confirmAppointment()` — bctrans entity 읽기 + 쓰기 + 블록체인 연동 원자성

### 7-2. `bctrans.common.enums` 위치 부적절
allcone 18파일 + common 10파일에서도 참조. Phase 1 common 분배 시 enum을 어디 두느냐에 따라 import 경로 전면 수정.

### 7-3. 배치/스케줄러 호출 경로 미확인
본 조사는 Controller 진입점 기준. `@Scheduled`, `@EventListener`로 호출되는 경로는 별도 확인 필요. dataLinkage, oldbctrans 폐기 결정 전 필수.

### 7-4. `AllconeVbsTruckTransOrderRepository`의 bctrans Mapper 직접 주입
3개 Mapper 주입: `ContainerTransportMapper`, `ContainerTransportVbsMapper`, `AllconeContainerTransportOrderMapper`. Phase 2에서 이 Repository를 HTTP bridge로 감쌀 때 각 호출 단위를 별도 API로 노출할지, Repository 단위로 감쌀지 설계 필요.

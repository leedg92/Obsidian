### 1. m_organization - 구조 단순화

- 변경: id(UUID) 제거하고 organization_id를 PK로 직접 사용
- 삭제: idx_m_organization_organization_id, idx_m_organization_active_yn 인덱스 제거
- 이유: 메타데이터 테이블(소규모)은 PK/UNIQUE만으로 충분. 추가 인덱스는 공간/성능 낭비

### 2. t_account - 인덱스 최적화 및 FK 정리

- 타입 변경: organization_id를 CHAR(36)에서 VARCHAR(50)으로 변경 (m_organization.organization_id와 타입 일치)
- 컬럼명 변경: role_type_id -> role_id
- FK 추가: role_id에 대한 외래키 제약 추가 (누락되어 있었음)
- 인덱스 제거:
    - idx_t_account_user_id (UNIQUE 제약이 이미 인덱스 역할)
    - idx_t_account_account_status (카디널리티 낮음. ACTIVE/INACTIVE 2개 값만 존재)
    - idx_t_account_deleted_at (soft delete 특성상 대부분 NULL이라 인덱스 효율 낮음)

### 3. h_login - 테이블 삭제

- 로그인 이력 테이블 전체 삭제
- 이유: 현재 요구사항에서 제외 (필요시 추후 추가)

### 4. m_role_types -> m_role - 테이블명 및 구조 변경

- 테이블명 변경: m_role_types -> m_role
- 컬럼명 변경: id -> role_id, role_type_name -> role_name
- 인덱스 제거: idx_m_role_types_role_type_name (UNIQUE가 이미 인덱스)
- 이유: 메타데이터 테이블은 불필요한 인덱스 제거

### 5. m_role_scopes - 컬럼명 변경

- 컬럼명 변경: id -> role_scope_id
- 인덱스 제거: idx_m_role_scopes_role_scope_name (UNIQUE가 이미 인덱스)
- 이유: 일관성 있는 네이밍, 불필요한 인덱스 제거

### 6. t_user_roles -> t_user_role_scopes - 구조 개선

- 테이블명 변경: t_user_roles -> t_user_role_scopes
- 역할 구조 변경:
    - 계정의 기본 역할은 t_account.role_id에 저장
    - 이 테이블은 추가 역할 범위만 관리
- PK 변경: id(UUID) 제거하고 (account_id, role_scope_id) 복합키를 PK로 사용
- 컬럼 제거: role_type_id 제거 (t_account에서 관리)
- 인덱스 최적화:
    - 기존 4개 인덱스 전부 제거
    - role_scope_id 단독 조회용 인덱스만 유지
    - 복합 PK가 (account_id, role_scope_id) 인덱스 역할 대체
- UNIQUE 제약 제거: 복합 PK가 유일성 보장

### 핵심 개선 포인트

1. 인덱스 최적화
    - 전체 인덱스 수: 12개 -> 6개
    - UNIQUE 제약이 있는 컬럼의 중복 인덱스 제거
    - 메타데이터 테이블의 불필요한 인덱스 제거
    - 카디널리티 낮은 컬럼(2-3개 값만 존재)의 인덱스 제거
2. 데이터 타입 일치
    - FK 관계의 컬럼 타입 통일 (organization_id: VARCHAR(50))
    - 누락된 FK 제약 추가 (t_account.role_id)
3. 자연키 활용
    - t_user_role_scopes에서 대리키 대신 복합 자연키 사용
    - UUID 생성 오버헤드 제거, 공간 절약
4. 네이밍 일관성
    - 테이블명과 컬럼명 명확화 (role_type_id -> role_id)

# allcone ↔ bctrans API 전수 조사 결과

> 작성일: 2026-04-17
> 브랜치: `feature/bctransApiForAllcone/WIP`
> 목적: 신 allcone이 bctrans를 HTTP로 호출할 때 필요한 API 전체 목록 확정

## 조사 범위

- allcone 패키지 Controller 27개 전수 확인
- bctrans/invoke/common 패키지의 serverName 분기 8개 Controller 전수 확인
- allcone → bctrans 방향: `import bctrans.*`, `${bctrans.url}`, `serverName` 분기
- bctrans → allcone 방향: `import allcone.*`, `${allcone.url}`, `serverName` 분기

## 1. allcone → bctrans 방향 (신 allcone이 호출해야 할 API)

### A. 이미 bctrans 인스턴스에 엔드포인트가 열려 있는 것 (15개)

**bctrans 인스턴스에 이미 동작하는 엔드포인트이므로 신규 작업 불필요.**

### C. serverName 분기 없고 엔드포인트도 없는 것 — 신규 API 필요 (VBS 예약 3개)

`VbsTruckTransOrderController`에서 serverName 분기 없이 `AllconeVbsTruckTransOrderService`를 직접 호출.
이 Service 내부에서 bctrans의 `ContainerTransportVbsDomainService`, `ContainerTransportDomainRepository`를 **내부 함수로 직접 호출**.
신 allcone이 HTTP로 호출하려면 bctrans에 wrapper API 신설 필요.

| #   | 엔드포인트                                             | 호출하는 bctrans 서비스                                                                                               | 종류    |
| --- | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ----- |
| 19  | `POST /mobile/trans/vbs/appointment/confirm`      | `ContainerTransportVbsDomainService.neededUpdateVbsForByConfirmAppointment()` + `.updateContainerVbsByPinNo()` | 읽기+쓰기 |
| 20  | `POST /mobile/trans/vbs/appointment/cancel`       | `AllconeVbsTruckTransOrderRepository` → bctrans Mapper 직접 호출 (`updateCancelAppointment`)                       | 쓰기    |
| 21  | `POST /mobile/trans/vbs/appointment/updateSingle` | `AllconeVbsTruckTransOrderRepository` → bctrans Mapper 직접 호출 (`updateSingleAppointment`)                       | 쓰기    |

### D. allcone Service가 bctrans Service/Repository/Mapper를 내부 함수로 직접 호출 — 신규 API 필요

#### D-1. PIN 중복 필터링 + DB 갱신 (복합 로직)
- **호출처**: `AllconeTransOrderDomainService.filteringVbsTransportOrderDataForPinNoDuplicate()`
- **호출처(2)**: `AllconeTransOrderDomainService.gateInTimeCorrectionDb()`
- **사용하는 bctrans 서비스**:
  - `ContainerTransportStatusDomainService.createNewCopinoGateInByPinNoDuplicate()`
  - `ContainerTransportSummaryDomainService.createContainerTransportSummaryForUpdate()`
  - `ContainerTransportSummaryDomainRepository.getContainerTransportSummaryByTransportCode()`
  - `ContainerTransportApplicationRepository.updateContainerTransportStatusAndSummary()`
- **권장**: 이 로직을 **bctrans에 통째로 이동**하고 단일 API로 제공
- **필요 API**: `POST /bridge/vbs/filter-pin-duplicate` (또는 적절한 경로)

#### D-2. TSS 상태 관련 (IttTruckTransStatusUpdateService 내부)
- `TssGroupOrderProcessService.saveGroupOrderTransStatus()` — 그룹오더 상태 저장
- `BctransIttTruckTransOrderService.updateTssOrderTransStatusMetrics()` — TSS 메트릭 업데이트
- `PushTruckerService.send()` — 운송사 시스템 알림 (FCM 아님, 외부 운송사 API 호출)
- **주의**: 이 3개는 `IttTruckTransStatusUpdateService.updateTransStatus()` 내부에서 호출됨. #5(`POST /mobile/trans/itt/status`)가 bctrans에서 이 서비스를 호출하는 구조라 **이미 엔드포인트 안에 포함**되어 있을 수 있음. 상세 확인 필요.

#### D-3. Mapper 직접 주입 (AllconeVbsTruckTransOrderRepository)
- bctrans Mapper 3개 직접 주입: `ContainerTransportMapper`, `ContainerTransportVbsMapper`, `AllconeContainerTransportOrderMapper`
- **사용 메서드**: `getAllconeTransportOrderDataByPinNo`, `getContainerTransportByPinNo`, `updateCancelAppointment`, `updateSingleAppointment`, `updateRequestAppointment`, `updateTruckTransStatusOnRoad`, `updateTruckTransOrderLatestStatusInZone`, `getContainerTransportVbsByTransportCode`
- **대부분 C그룹(예약 3개)과 A그룹(status/cancelStart/blockin)에서 호출됨**
- A그룹은 이미 HTTP 크로스콜 → Mapper 호출은 bctrans 인스턴스 내부에서 일어남 → **추가 API 불필요**
- C그룹(예약)은 신규 API 필요 (위 #19~#21)

### E. API 불필요 — allcone 자체 데이터 / bctrans 무관

| 엔드포인트 | Controller | 이유 |
|-----------|-----------|------|
| `GET /mobile/trans/order/list` | VbsTransOrderController | allcone Mapper(`VbsTransOrderMapper`, `TssTransOrderMapper`) 사용 |
| `GET /mobile/trans/order/eslip` | VbsTransOrderController | 동일 |
| `POST /mobile/trans/order/remove` | VbsTransOrderController | 동일 |
| `POST /mobile/trans/history` | TruckTransOrderController | `common.trans.service.TransHistoryService` |
| `POST /mobile/trans/itt/history` | IttTruckTransOrderController | `AllconeIttTruckTransOrderService` (allcone 자체) |
| `POST /mobile/eslip/getUserPosition` | EslipDataController | allcone 자체 서비스 |
| `POST /mobile/terminal/req/gateout` | TerminalRequestApiController | serverName 분기 없음, `TerminalRequestService.requestTerminalGateOut()` 직접 호출 (터미널 직접 통신) |
| `POST /mobile/terminal/req/changeConDoorDirection` | TerminalRequestApiController | serverName 분기 없음, 터미널 직접 호출 |
| 회원/로그인/알림설정 등 | SignController, UserInfoController, UserTruckController, AlarmConfigController 등 (12개) | allcone 전용 DB, bctrans 무관 |

**단, Order/Eslip 조회(list, eslip)가 쿼리하는 테이블이 allcone DB인지 bctrans DB인지 확인 필요.**
allcone Mapper를 사용하지만 같은 물리 DB를 공유하고 있어서, 신 allcone이 별도 DB를 갖게 되면 이 조회 API도 bctrans에서 열어줘야 할 수 있음.

## 2. bctrans → allcone 방향 (역방향)

### HTTP 호출 — 2건 (FCM 위임만)

| # | 호출 코드 | 호출 URL | 목적 |
|---|---------|---------|------|
| 1 | `BctransVbsTruckTransOrderService.sendVbsAlarm()` | `${allcone.url}/vbs/invoke/alarm` | VBS 이벤트 → allcone FCM 푸시 위임 |
| 2 | `BctransIttTruckTransOrderService.sendTssAlarm()` | `${allcone.url}/itt/invoke/alarm` | TSS 이벤트 → allcone FCM 푸시 위임 |

**결론**: 역방향 HTTP 호출은 FCM 위임 2건뿐. 신 allcone이 `/vbs/invoke/alarm`, `/itt/invoke/alarm` 동일 스펙으로 열면 `allcone.url` 프로퍼티만 변경하면 됨.

### 내부 함수 참조 (import) — VO/DTO 참조 + allcone 프로필 전용

#### bctrans 패키지 → allcone (4개 파일, 5건)
| 파일 | import | 성격 |
|------|--------|------|
| `BctransVbsTruckTransOrderService` | `AllconeVbsTruckTransOrderService`, `BctransDigitalGateinRequest` | 디지털게이트인 처리, allcone 서비스 호출 |
| `ContainerTransportStatusDomainService` | `VbsOrderData` | VO 타입 참조만 |
| `BctransIttTruckTransOrderService` | `TransshipmentTransStatus` | VO 타입 참조만 |
| `SecurityConfig` | `FcmDeviceTokenRepository` | FCM 디바이스 토큰 (allcone DB) |

#### invoke 패키지 → allcone (6개 파일, 13건)
| 파일 | import | 성격 |
|------|--------|------|
| `VbsInvokeAlarmController` | `AllconeVbsTruckTransOrderService` | serverName="allcone" 분기에서만 사용 |
| `VbsInvokeAlarmService` | `DeviceTokenService`, `TempAdminUserTruckService` | allcone 프로필 전용 FCM 처리 |
| `IttInvokeAlarmService` | `DeviceTokenService`, `TempAdminUserTruckService`, `AllconeIttTruckTransOrderService` | 동일 |
| `NoticeInvokeAlarmService` | `AlarmService`, `AlarmMessageData`, `AlarmTargetData` | allcone 프로필 전용 공지 알림 |
| `InvokeService` | `AlarmConfig`, `DeviceTokenService` | allcone 프로필 전용 알림 설정 |
| `InvokeRepository` | `AlarmConfigMapper`, `AlarmConfig` | allcone 프로필 전용 알림 설정 DB |

#### common 패키지 → allcone (10개 파일, 17건)
- 대부분 VO/DTO 타입 참조 (`EslipDataOld`, `VBSEslipData`, `InternalGateData`, `GetTransHistoryListRequestDto` 등)
- `FcmDeviceTokenRepository`, `DeviceTokenService`: FCM 관련
- `EslipDataMapper`: eslip 데이터 조회

**결론**: 역방향은 **HTTP 2건(FCM) + VO/DTO 참조 + allcone 프로필 전용 invoke 로직**. bctrans 프로필로만 기동하면 invoke의 allcone 분기는 실행되지 않으므로, 실질적 역방향 런타임 의존은 FCM HTTP 2건뿐.

## 5. 최종 작업 목록

### 즉시 작업 가능 (bctrans에 신규 API 추가)
1. `POST /bridge/vbs/appointment/confirm` — VBS 예약 확정 wrapper
2. `POST /bridge/vbs/appointment/cancel` — VBS 예약 취소 wrapper
3. `POST /bridge/vbs/appointment/updateSingle` — VBS 예약 수정 wrapper
4. `POST /bridge/vbs/filter-pin-duplicate` — PIN 중복 필터링 (로직 이동)

### 확인 후 작업 여부 결정
5. D-2 TSS 상태 관련 3건 — #5(`/mobile/trans/itt/status`)의 bctrans 동작 범위에 이미 포함인지 확인
6. `POST /mobile/terminal/req/gateout` — allcone 개발자는 필요하다고 했지만 코드상 serverName 분기 없음 → 재확인
7. Order/Eslip 조회 — 테이블 소유가 allcone인지 bctrans인지에 따라 결정

### 작업 불필요
8. A그룹 15개 + B그룹 3개 = 18개 — 이미 열려 있음

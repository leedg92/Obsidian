# allcone-bctrans 분리 작업: 방향성 및 이슈 정리

> 작성일: 2026-04-17
> 상태: 논의 중 (allcone 개발자와 합의 필요)
> 관련: [MSA 아키텍처 초안](msa-architecture-draft.md), [MSA 점진적 전환 계획](msa-migration-plan.md)

## 배경

- 신 allcone: 다른 개발자가 새 프로젝트 + 새 스택으로 처음부터 구축 중
- 기존 프로젝트: 같은 소스코드를 Spring 프로필(allcone/bctrans)로 분기 배포하는 모노리스 구조
- bctrans는 코어 로직이 많아 당분간 현행 스택 유지

## 핵심 결론: 양방향 통신 구조

코드 분석 결과, **양방향 모두 이미 HTTP 통신 구조**임이 확인됨:

### bctrans → allcone (FCM 푸시 위임) — 이미 HTTP
```
외부 이벤트 → bctrans(invoke, serverName="bctrans")
           → bctransVbsTruckTransOrderService.sendVbsAlarm()
           → HTTP POST: ${allcone.url}/vbs/invoke/alarm  ← allcone 인스턴스로 HTTP 전달
           → allcone 인스턴스(serverName="allcone")
           → VbsInvokeAlarmService → FCMNotificationService (직접 FCM 전송)
```
- FCM 직접 전송은 allcone 프로필에서만 수행
- bctrans 프로필은 `allcone.url` 프로퍼티 기반 HTTP 호출로 위임
- **따라서 역방향은 신경 쓸 필요 없음** — 신 allcone이 동일 invoke 엔드포인트를 열면 `allcone.url`만 변경하면 됨

### allcone → bctrans — 현재 혼재 (내부함수 + HTTP)
- 일부는 이미 HTTP (`${bctrans.url}/mobile/trans/...` 형태, 7개 존재)
- 일부는 내부 함수 직접 호출 (bctrans Service/Repository 직접 주입)
- **이 방향만 API를 열어주면 됨**

### 결론
> **bctrans에 allcone→bctrans 방향 API만 추가하면, 구 bctrans(+API) 버전을 신/구 allcone 양쪽에서 동시 사용 가능**
> - 구 allcone: 기존 내부 함수 호출 (같은 WAR)
> - 신 allcone: 새로 열린 API를 HTTP로 호출
> - bctrans → allcone(FCM): 이미 HTTP이므로 `allcone.url` 프로퍼티만 변경

## 전체 로드맵 (allcone 기준)

| 단계 | 형태 | 설명 |
|------|------|------|
| **1차 (현재)** | 앱 → 구 allcone → bctrans(내부함수) + 신 allcone(DB 동기화만) | 신 allcone은 데이터 동기화만, bctrans API 불필요 |
| **2차** | 앱 → 구 allcone → bctrans(내부함수) + 앱 → 신 allcone → bctrans(HTTP API) | 신/구 동시 운영, bctrans에 API 추가 필요 |
| **최종 (allcone 기준)** | 앱 → 신 allcone → bctrans(HTTP API) | 구 allcone 제거, 모두 API 통신 |
| **이후 (bctrans 기준)** | bctrans도 신스택 적용 + 리팩토링 | 열어놓은 API를 기준으로 전환 |

## bctrans 입장에서의 작업

### 지금 해야 할 일 (1차 → 2차 준비)
- **allcone → bctrans 방향 API만 추가** (기존 코드 유지, allcone 패키지 건드리지 않음)
- 구 bctrans(+API) 버전 하나로 신/구 allcone 모두 대응

### 최종 단계 이후
- 구 allcone 내린 후 allcone 패키지 제거 가능
- serverName 분기 로직 제거 (bctrans 고정)
- bctrans 신스택 전환

## allcone 개발자가 정리한 API 목록

### Terminal
- 진입점: `TerminalRequestApiController`, `ContainerHandlingBcRequestService`
- `POST /mobile/terminal/req/gatein`
- `POST /mobile/terminal/req/gateout`
- `POST /mobile/terminal/req/copino/verify`
- `POST /mobile/terminal/req/internal/gatein`
- `POST /mobile/terminal/req/container/handle/changeDirection`
- `POST /mobile/terminal/req/container/handle/swap`
- `POST /mobile/terminal/req/container/handle/swap/reason`

### TSS
- 진입점: `IttTruckTransOrderController`, `IttTruckTransStatusUpdateService`
- `POST /mobile/trans/itt/status`
- `POST /mobile/trans/itt/cancelStart`
- `POST /mobile/trans/itt/req/blockin`
- 이력 조회 API도 원천 데이터가 bctrans면 추가 검토

### VBS
- 진입점: `VbsTruckTransOrderController`, `AllconeVbsTruckTransOrderService`
- `POST /mobile/trans/vbs/status`
- `POST /mobile/trans/vbs/cancelStart`
- `POST /mobile/trans/vbs/req/blockin`
- `POST /mobile/trans/vbs/req/digitalGatein`
- 예약 확정/취소/수정도 원천 데이터가 bctrans면 추가 검토

### Order / Eslip
- bctrans enum, 엔티티, 리포지토리에 직접 의존 많음
- bctrans가 order, eslip 조회 API 제공 필요
- allcone은 화면 조합만 수행

### 권장 namespace
- `/api/v1/allcone/terminal/*`
- `/api/v1/allcone/tss/*`
- `/api/v1/allcone/vbs/*`
- `/api/v1/allcone/order/*`

## 기존 분석(bctrans 측)과의 비교

기존 분석: `docs/bctrans-api-for-allcone-analysis.md` (2026-04-15)

| 영역 | 기존 분석 | allcone 개발자 분석 | 차이 |
|------|----------|-------------------|------|
| Terminal | 포함됨 | 7개 명시 | 거의 일치 |
| TSS | 3개 | 3개 + 이력조회 가능성 | 거의 일치 |
| VBS | 7개 (내부함수 호출 기준) | 4개 + 예약 관련 추가 가능성 | 범위 재확인 필요 |
| Order/Eslip | **없었음** | 조회 API 필요하다고 명시 | 추가 항목 |
| 경로 구조 | `/bridge/*` (단일 Controller) | `/api/v1/allcone/{도메인}/*` (도메인별 분리) | 구조 다름 |

### 추가 확인 필요 사항
- VBS 예약 확정/취소/수정: 원천 데이터 위치에 따라 API화 여부 결정
- TSS 이력 조회: 동일
- Order/Eslip 조회: 기존 분석에 없었으므로 범위 추가 검토
- 전체 대상 촘촘한 분석 미완료 (serverName 분기 + 내부함수 호출 전수 조사 진행 중)

## 이슈

### ~~ISSUE-1: FCM 푸시 역방향 호출 문제~~ (해소)

**결론: 이슈 아님.**
- bctrans → allcone FCM 위임은 이미 HTTP 통신 (`${allcone.url}/vbs/invoke/alarm`)
- 신 allcone이 동일 invoke 엔드포인트를 열면 `allcone.url` 프로퍼티만 변경하면 됨
- L4 라우팅 문제 없음 (bctrans가 직접 allcone URL을 지정하므로)

**신 allcone 개발자 확인 필요:**
- 신 allcone이 `/vbs/invoke/alarm`, `/itt/invoke/alarm` 등 invoke 엔드포인트를 동일 스펙으로 열 수 있는지

### ~~ISSUE-2: 신/구 동시 운영 시 인스턴스 그룹 분리 방안~~ (해소)

**결론: 이슈 아님.**
- 구 bctrans(+API 추가) 버전 하나로 양쪽 대응 가능
- 구 allcone: 같은 WAR 내부 함수 호출 (기존 그대로)
- 신 allcone: 새로 추가된 API를 HTTP 호출
- 인스턴스 그룹 분리 불필요

### ISSUE-3: API 대상 전수 조사 미완료

**상태: 진행 중**

- allcone 개발자 목록과 bctrans 측 분석이 완전히 일치하지 않음
- serverName 분기 17개 메서드 + 내부 함수 호출 전수 조사가 아직 진행 중
- Order/Eslip 영역이 기존 분석에서 빠져 있었음

**다음 작업:**
- allcone → bctrans 직접 참조 전수 조사 완료
- serverName 분기 상세 분석 완료
- 두 분석 결과를 allcone 개발자 목록과 대조
- 최종 API 목록 확정

### ISSUE-4: API 경로 구조 미확정

**선택지:**

| 방식 | 장점 | 단점 |
|------|------|------|
| 기존 경로 유지 (`/mobile/trans/vbs/status` 등) | 이미 존재, 변경 없음 | allcone 전용인지 구분 안 됨 |
| 별도 namespace (`/api/v1/allcone/*`) | 명확한 분리, 내부 모델 은닉 | 새로 만들어야 함, 기존과 중복 |

**참고:** 이미 allcone 코드 내에서 bctrans를 HTTP 호출하는 패턴이 7개 존재 (기존 `${bctrans.url}/mobile/trans/...` 형태)

### ISSUE-5: 인증/보안 방식 미확정

- 현재 allcone ↔ bctrans는 같은 WAR이라 인증 없음
- 신 allcone이 HTTP로 호출 시 인증 필요 여부
- 후보: 내부 토큰 / IP 화이트리스트 / API Key / 없음(내부망 신뢰)

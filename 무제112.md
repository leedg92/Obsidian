  여기의 예시 건은

  - A : DYS_SSNJ2600844_DYLU5127231 (blNo 있음)
  - B : DYS_-_DYLU5127231 (blNo 없음)

  1. 같은 운송건의 운송오더가 중복 생성 (사업자번호 다름 - 운송사 두 곳에서 쏜 듯)
  2. 이후 blockchain이 두 건 모두의 "반출 이벤트"(60/70/80)를 bctrans에 내려줌 (A/B)
  3. "반입 이벤트"(90/100/110)는 B에만 내려줌 → B는 정상 완료, A는 반출 80에서 멈춤(미완료)
  4. 다음날 00:09 A건에 GateOut(110, gateType=CALL_CENTER) 트리거 발생
      1. 콜센터가 A건을 GateOut으로 찍음 → 페이로드엔 dispatchGroup이 빈값이라 단건 경로
      2. bctrans는 A건의 docKey로 조회 → A건만 UPDATE
         (TRANS_STAT=110, TRANS_STAT_DT=06/17 00:09 로 갱신, MOD_DT도 갱신)
      3. 앱 노출은 1차로 "REG_DT > (현재-1일) AND TRUCK_NO"로 긁지만(A건 REG_DT 06-16 14:14 → 범위 안),
         실제 표시 여부는 categorizeTssOrders가 "transStatusTime(TRANS_STAT_DT)이 최근 6시간 이내인지"로 결정
         → 완료(110)인지 아닌지는 안 따짐. 6시간 이내면 완료 건도 노출
      4. A건은 14:47(반출80)에서 멈춰 ~20:47에 6시간 경과로 화면에서 사라졌으나,
         콜센터의 110이 TRANS_STAT_DT를 00:09로 갱신 → 6시간 타이머가 리셋
         → 사라졌던 A건이 다시 노출됨 (= "완료된 운송오더가 새벽에 되살아남" 민원의 실체)

  정리하자면, 노출/숨김 기준은 "완료 여부"가 아니라 "transStatusTime이 최근 6시간 이내인가"이고, 콜센터 GateOut이 그 시각을 갱신해 타이머를 리셋한 것이 부활의 직접 원인.

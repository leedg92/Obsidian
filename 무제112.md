# allcone ↔ bctrans 분리 API 최종 명세

> 이 문서는 신 allcone(별도 프로젝트·별도 스택) 분리 시 양쪽이 열어야 할 HTTP API의 **단일 진실(single source of truth)** 문서다.
> 분석 중 생성된 보조 문서(`docs/msa-*.md`, `docs/package-*.md`, `docs/allcone-api-mapping-verification.md` 등)가 아닌, **결론만** 담는 한 파일로 관리한다.
> 수치·경로가 변경되면 반드시 이 파일을 갱신한다.
>
> 최종 갱신: 2026-04-22 (분석자 3명 + 검토자 루프 후 확정)
> 브랜치: `feature/bctransApiForAllcone/WIP`

---

## 1. 배경

- 기존 구조: `allconebctrans` 단일 WAR를 Spring 프로필(`allcone`/`bctrans`)로 분기 배포
- 신 allcone: 다른 개발자(가연)가 새 프로젝트·새 스택으로 처음부터 구축 중
- 구 allcone은 현행 유지, 신 allcone 분리 시점에 양쪽이 열어야 할 API를 확정하는 것이 본 문서의 목적
- 이 브랜치 원칙: **기존 Service/Repository/Entity 로직 변경 금지, 얇은 wrapper Controller/DTO만 추가**

---

## 1-2. 테이블 소유 규칙

현재 물리 DB는 `bctransdbx` 단일 스키마. 테이블명으로 소유를 구분.

| 소유 | prefix / 예외 | 비고 |
|------|--------------|------|
| **allcone** | `allcone_*` | `allcone_terminal_*` 포함 (prefix는 terminal이 아닌 allcone) |
| **allcone** | `user_regist_truck` | prefix 예외 — 앱 회원 차량. Mapper 위치 `allcone/user/UserTruckMapper.xml` 등 |
| **bctrans**(논리) / allcone(파일) | `user_digital_gate_request` | **논리 소유는 bctrans**(컬럼이 모두 운송 이벤트 데이터, `userId` 없음). INSERT 주체도 bctrans facade D7로 이동 예정. **파일 위치는 allcone 유지**(`allcone/geofence/`, `resources/mappers/mainDataSource/allcone/digitalGate/`), Phase 2에서 bctrans로 이동. 현 브랜치에서는 facade(§2-1 D7)가 기존 allcone Repository를 Bean 주입해 INSERT 수행 |
| **bctrans** | `general_*`, `terminal_*`(allcone_* 제외), `container_*`, `tb_b_*`, `bctrans_*`, `tss_*` | 운송 도메인 전반 |
| **외부** | `chainportaldb.*` | 외부 DB (공지, 이벤트 등) — allcone가 직접 접근 허용 |

### 판정 방법
- `src/main/resources/mappers/` Mapper XML의 `FROM`/`JOIN` 테이블 확인
- **Mapper 파일이 놓인 경로(allcone/ / bctrans/ / common/)와 실제 테이블 소유는 별개 개념**
  - 예: `allcone/geofence/GeofenceMapper.xml`이 `terminal_area`(bctrans 소유) 조회 → "침범 B"
  - 이런 침범을 해소하기 위해 본 문서의 신규 API(D/β/γ)가 필요

#### D7 상세 (디지털 게이트인 요청 pass-through facade)

**요청**
```
POST /bridge/vbs/req/digitalGatein
Headers: deviceToken (서버 간 인증 방식에 따라 조정)
Body: { pinNo, truckNo, headingDirection, latitude, longitude }   // 앱 원본 그대로
```

**bctrans facade 내부 처리 (얇은 wrapper, 파일 이동 없음)**
1. `ContainerTransportDomainRepository.getContainerTransportByPinNo(pinNo)` (bctrans 내부 호출, HTTP 불필요)
   - null → `ApiResponse.error("Bad Request")` 조기 반환
2. `TextCryptoService.encrypt(latitude/longitude)` — bctrans facade에서 암호화 수행 (`common/support/crypt/TextCryptoService.java`는 common이라 양쪽 주입 가능)
3. `UserDigitalGateInRequestEntity.fromEDIData(...)` 조립 (기존 allcone Entity 그대로 재사용)
4. `BctransDigitalGateinRequest` 조립 (requestSeq=UUID.randomUUID(), gateInTime, docKey/conNo/terminalCode)
5. 기존 `BctransVbsTruckTransOrderService.requestDigitalGatein(BctransDigitalGateinRequest)` 호출 — 외부 터미널 API 포워딩 재사용
6. `DigitalGateRepository.insertUserDigitalGateInRequestEntity(entity)` — 기존 allcone Repository를 Bean 주입해 INSERT (기존 bridge 7개와 동일한 패턴)
7. `return ApiResponse`

**배경**
- 앱은 `pinNo`만 아는데 터미널 API 호출에는 `terminalCode`/`docKey`/`conNo`가 필요. 원래 allcone가 bctrans DB를 조회해 파생값을 만들어 bctrans로 전달했지만, 신 allcone 분리 후에는 이 조회 자체가 HTTP 왕복이 됨
- facade를 bctrans에 두면 조회가 bctrans 내부 호출로 처리되어 `/bridge/vbs/container-transport?pinNo=` 같은 추가 조회 API 불필요
- 구 allcone은 본 facade를 사용하지 않으므로 `AllconeVbsTruckTransOrderService.requestDigitalGatein` 로직 변경 없음 → **1단계에서 기존 코드 이동·수정 없음**
- `user_digital_gate_request` 테이블의 물리 파일 이동(Entity/Mapper/Repository)은 Phase 2에서 수행

## 4. 앱(app-allcone) 측 사실

- 앱 위치: `/mnt/d/workspace/bctrans_prod/app-allcone` (Flutter)
- 앱이 호출하는 모든 URL은 `/allcone-api` 프리픽스로 통합. `api-bctrans` 별도 호출 경로는 **없음** → 앱은 allcone과만 통신, 내부에서 allcone가 bctrans로 위임
- 판별 규칙: 경로에 `api-allcone` 포함 = allcone 소통 API, `api-bctrans` = bctrans 소통, 기타 도메인(chainportal 등) = 외부
- 앱 RemoteConfig 기본값 기준 URL 수: 83~85개 (GPS 2건은 RemoteConfig 외 경로일 수 있어 추가 확인 영역)

## 5. 미결정 사항 (가연과 합의 필요)

1. **deviceToken 헤더 포함 여부**: 기존 Controller는 `@RequestHeader("deviceToken")`로 모바일 기기 토큰 검증. 본 bridge는 서버 간 호출이라 현재 뺐는데, 신 allcone가 보낼지 미정.
2. **응답 형식 일관성**: bridge 중 `/bridge/order/list`, `/bridge/order/eslip`은 `ResponseEntity<T>` 직접 반환, 나머지는 `ApiResponse` 래핑 → 통일 여부 미정.
3. **인증 방식**: `/bridge/*`, `/allcone-internal/*` 모두 현재 인증 없음. JWT / IP allowlist / 내부망 신뢰 / API Key 중 선택 필요.

## 6. 작업 순서

1. 미결정 3건(§5) 결정
2. bctrans 측 신규 6개(D1~D6) facade 구현 — `bctrans/bridge/web/AllconeBridgeController.java` 확장
3. allcone 측 신규 6개(β2·β3·β4·β5·β6·γ1) 스펙 가연에게 공유, 신 allcone가 구현
4. 통합 테스트: bctrans 인스턴스가 신 allcone을 호출하는 경로(`${allcone.url}` 교체)와 신 allcone가 bctrans를 호출하는 경로(`/bridge/*`) 양방향 동일성 검증

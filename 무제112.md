# 패키지 재배치 계획 (Phase 2 상세)

> 작성일: 2026-04-20
> 브랜치: `feature/bctransApiForAllcone/WIP`
> 목적: MSA Phase 2 — common 분배 후 남는 침범 해소. "각 패키지에 무엇을 두고, 무엇을 API로 열지" 확정

## 전제

1. **common은 양쪽에 복사 후 unused 정리** (Phase 1)
2. **common 복사로 자동 해소되는 것**:
   - `bctrans.common.enums` 참조 (40+건)
   - `common.tss/`, `common.vbs/` 도메인
   - VO import (`VbsOrderData`, `TransshipmentTransStatus`, `TSSEslipData` 등)
3. **common 복사로 해소 안 되는 것** = 본 문서 대상

## 1. "allcone 패키지에 있지만 bctrans 영역" — bctrans 이동 대상

allcone 네이밍이 붙었을 뿐 실제로는 bctrans 테이블만 다루는 서비스. Phase 2에서 bctrans 패키지로 이동하면 역방향 침범이 자동 해소된다.

### 1-1. 순수 bctrans 영역 (즉시 이동 가능)

| 현재 위치 | 주석 / 용도 | 사용 테이블 | 이동 후 |
|---------|----------|-----------|--------|
| `allcone/order/service/VbsTransOrderService` | 운송 오더 목록/E-Slip/삭제 | `container_transport*`, `tb_b_*`, `terminal_truck_event` | `bctrans/order/service/` |
| `allcone/order/service/AllconeTransOrderDomainService` | PIN 중복 필터링, gateIn 시간 보정 | bctrans Service 5개 + Repository | `bctrans/order/service/` |
| `allcone/vbs/repository/AllconeVbsTruckTransOrderRepository` | bctrans Mapper 3개 주입 | `container_transport*` | `bctrans/vbs/repository/` |
| `allcone/tss/service/AllconeIttTruckTransOrderService` | dispatchInfoId 조회 + 운송이력 | `tb_b_truck_trans_odr*` | `bctrans/tss/service/` |
| `allcone/tss/service/IttTruckTransStatusUpdateService` | ITT 상태 변경 | `tb_b_*` | `bctrans/tss/service/` |
| `allcone/order/mapper/VbsTransOrderMapper` | SQL 전체 bctrans | 〃 | `bctrans/order/mapper/` |
| `allcone/order/mapper/TssTransOrderMapper` | SQL 전체 bctrans | 〃 | `bctrans/tss/mapper/` |
| `allcone/order/mapper/GateInWithOutCopinoMapper` | terminal_truck_event | 〃 | `bctrans/order/mapper/` |
| `allcone/eslip/mapper/EslipDataMapper` | container_transport | 〃 | `bctrans/eslip/mapper/` |
| `allcone/containerHandlingRequest/mapper/ContainerHandlerReasonMapper` | bctrans_container_handler_reason | 〃 | `bctrans/containerHandlingRequest/mapper/` |

관련 VO/Entity도 함께 이동.

| 현재 위치 | 설명 | 분리 방식 |
|---------|------|---------|
| `allcone/vbs/service/AllconeVbsTruckTransOrderService` | 주석에 `[HYBRID]` 명시. VBS 예약/상태/디지털게이트인 | **메서드 단위 분리** — 아래 표 참조 |

## 2. bctrans가 열어야 할 Bridge API (allcone → bctrans 침범 해소)

### 2-1. facade 가능 (단순 위임, 신규 6개)

앞서 `allcone-api-mapping-verification.md` 3.2절에서 확정:

| # | 경로 | 설명 |
|---|------|------|
| 1 | `GET /bridge/terminal/list` | 터미널 목록 |
| 2 | `GET /bridge/terminal/info` | 터미널 상세 + area |
| 3 | `GET /bridge/terminal/contract/list` | 계약 터미널 |
| 4 | `GET /bridge/geofence/digitalGateArea` | 디지털게이트 polygon |
| 5 | `POST /bridge/trans/history` | VBS 운송 이력 |
| 6 | `POST /bridge/trans/itt/history` | ITT 운송 이력 |

### 2-2. facade 불가 (비즈니스 로직 포함, 기존 7개)

이미 구현됨. 운송 로직이 bctrans 테이블 + 블록체인 + 파생 계산이 엮여 있어 순수 위임 불가.

| # | 경로 | 설명 |
|---|------|------|
| 1 | `POST /bridge/vbs/appointment/confirm` | 예약 확정 (블록체인 + DB) |
| 2 | `POST /bridge/vbs/appointment/cancel` | 예약 취소 (블록체인 + DB) |
| 3 | `POST /bridge/vbs/appointment/updateSingle` | 예약 변경 (블록체인 + DB) |
| 4 | `POST /bridge/vbs/filter-pin-duplicate` | PIN 중복 필터링 (bctrans Service 5개 조합) |
| 5 | `GET /bridge/order/list` | 주문 목록 (다중 테이블 조인 + 가공) |
| 6 | `GET /bridge/order/eslip` | E-Slip 목록 |
| 7 | `POST /bridge/order/remove` | 주문 삭제 + FCM |

### 2-3. 이미 HTTP 통신 중 (그대로 재사용, 17개)

`allcone-api-mapping-verification.md` 3.1절 참조. 신규 작업 불필요.

## 3. allcone가 열어야 할 Bridge API (bctrans → allcone 침범 해소)

### 3-1. invoke 패키지(bctrans 소속)가 호출 중인 allcone 고유 메서드

| 호출 위치 | 메서드 | 용도 | 제안 API |
|---------|-------|------|---------|
| `VbsInvokeAlarmService`, `IttInvokeAlarmService`, `InvokeService` | `DeviceTokenService.getUserIdByTruckNoV2(truckNo)` | 차량번호 → userId | `GET /allcone-internal/device-token/user-id?truckNo=` |
| `VbsInvokeAlarmService`, `IttInvokeAlarmService` | `DeviceTokenService.getDeviceTokenByTruckNoV2(truckNo)` | 차량번호 → FCM 토큰 | `GET /allcone-internal/device-token?truckNo=` |
| `VbsInvokeAlarmService`, `IttInvokeAlarmService` | `TempAdminUserTruckService.getAdminTokenData(truckNo)` | 관리자 토큰 리스트 (인메모리) | `GET /allcone-internal/admin-token?truckNo=` |
| `NoticeInvokeAlarmService` | `AlarmService.getEmergencyMessageBySeq(seq)` | 알람 메시지 조회 | `GET /allcone-internal/alarm/message?seq=` |
| `NoticeInvokeAlarmService` | `AlarmService.pickOutInvokeSendersV2(seq)` | 알람 대상자 조회 | `GET /allcone-internal/alarm/targets?seq=` |
| `NoticeInvokeAlarmService` | `AlarmService.setAlarmPushSuccess(seq)` | 알람 푸시 성공 마킹 | `POST /allcone-internal/alarm/push-success` |
| `InvokeService`, `InvokeRepository` | `AlarmConfigMapper` (AlarmConfig 조회) | 사용자 알람 설정 | `GET /allcone-internal/alarm-config?userId=` |

**프리픽스 `/allcone-internal/`**: 서버 간 내부 통신 (앱 호출 아님). 현재 `${allcone.url}/vbs/invoke/alarm`, `/itt/invoke/alarm` 2건과 구분.

### 3-2. bctrans 패키지(vbs/tss)가 호출 중인 allcone 메서드

| 호출 위치 | 메서드 | 처리 |
|---------|-------|------|
| `BctransVbsTruckTransOrderService:345` → `AllconeVbsTruckTransOrderService.cancelAppointment` | VBS 이벤트 수신 후 예약 취소 | **1-2절에서 cancelAppointment가 bctrans 이동 대상** → 동일 패키지 내 호출로 자동 해소 |
| `BctransVbsTruckTransOrderService` → `BctransDigitalGateinRequest` (DTO) | 디지털게이트인 요청 DTO | **DTO 복사** (VO 공유 해결) |

### 3-3. SecurityConfig의 allcone Repository 주입

| 호출 위치 | 메서드 | 처리 |
|---------|-------|------|
| `bctrans/auth/infrastructure/config/SecurityConfig` → `FcmDeviceTokenRepository` | 인증 필터에서 FCM 토큰 확인 | **SecurityConfig은 allcone 프로필에서만 필요** → bctrans 프로필에선 다른 설정 사용, 또는 분기 제거 후 bctrans 측 인증 로직 분리 |

## 4. 최종 그림 (Phase 2 완료 후)

### allcone 패키지 (축소)
- 앱 전용: 회원/FCM/알림/차량/안전규정/앱설정 Controller + Service + Repository
- `gps`, `insa` 포함
- 신규 API: **Bridge API 7개** (3-1 절 기준)
- `user_regist_truck`, `user_digital_gate_request`, `allcone_*` 테이블 소유

### bctrans 패키지 (확장)
- 운송 도메인: VBS, TSS, 터미널, e-Slip, 컨테이너 핸들링
- 1-1절 이동 대상 흡수
- 1-2절 분리된 bctrans 영역 흡수
- `invoke/` (handler 전신)
- 신규 API: **Bridge API 6개 facade** (2-1 절 기준)
- 기존 bridge 7개 유지 (2-2 절)
- 기존 HTTP 17개 재사용 (2-3 절)
- `container_transport*`, `general_*`, `terminal_*`, `tb_b_*` 등 테이블 소유

### common 패키지 (소멸)
- allcone/bctrans 양쪽으로 복사 후 삭제

### 폐기 패키지
- `oldbctrans`, `dataLinkage` (사용 재확인 후)

## 5. 작업 순서 제안

1. **Phase 1 (common 분배)** — 먼저 완료하여 자동 해소될 것 제거
2. **bctrans 이동 (1-1)** — allcone 네이밍 서비스를 bctrans로 이동 (순수 bctrans 영역)
3. **메서드 단위 분리 (1-2)** — AllconeVbsTruckTransOrderService 분리
4. **allcone Bridge API 신규 구축 (3-1)** — invoke가 HTTP로 호출하도록 전환
5. **bctrans Bridge API facade 6개 신규 구축 (2-1)**
6. **HTTP 호출 래퍼 제거** — `*RequestBctrans`, `*ToBctrans` 메서드 삭제
7. **SecurityConfig 분리 (3-3)** — 프로필별 인증 설정 분리

## 6. 미확정 / 추가 검토

1. **`requestDigitalGatein`의 분리 방식**: `user_digital_gate_request` INSERT(allcone)와 bctrans 이벤트 전달이 같은 트랜잭션에 묶여 있음. 분산 처리 시 원자성 보장 방식 결정 필요.
2. **Bridge API 인증**: `/bridge/*`, `/allcone-internal/*` 내부 전용이지만 인증 방식 미확정.
3. **batch / scheduler 호출**: 본 분석에 포함 안 됨. Phase 2 시작 전 재확인 필요.
4. **AlarmConfigMapper 직접 주입**: `InvokeRepository`가 allcone Mapper 직접 주입 → 3-1 API 전환 시 조회 외 write 호출 여부 확인 필요.

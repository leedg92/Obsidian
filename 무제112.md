## 1. 개요

AllconeApp(모바일)에서 NICE 본인인증(Unified Auth V2)을 도입하고, 기존 회원 테이블 `tb_a_user_m`을 포털 웹과 공용인 `tb_member`로 전환하는 작업입니다.

**핵심 변경 사항:**
- NICE 본인인증 연동 모듈 신규 개발
- CI(개인식별번호) 양방향 암호화 + 단방향 해시 체계 도입
- 전 Mapper에서 `tb_a_user_m` → `tb_member` 테이블 참조 변경
- 로그인 실패 카운트 별도 테이블(`tb_a_user_login_err_i`) → `tb_member` 컬럼으로 통합

### 2.2 NICE 서버 통신 구조

| 항목 | 내용 |
|------|------|
| **NICE API Base URL** | `https://auth.niceid.co.kr` |
| **인증 방식** | Bearer Token (access_token) |
| **토큰 발급** | Basic Auth (Base64(client_id:client_secret)), grant_type=client_credentials |
| **요청 헤더** | `Authorization: Bearer {token}`, `X-Intc-DevLang: linux/java`, `Content-Type: application/json` |
| **API 버전** | Unified Auth V2 (PBKDF2 기반 키 유도) |

**사용 API 엔드포인트:**

| 엔드포인트 | 메서드 | 용도 |
|-----------|--------|------|
| `/ido/intc/v1.0/auth/token` | POST | 액세스 토큰 발급 |
| `/ido/intc/v1.0/auth/url` | POST | 인증 창 URL 생성 |
| `/ido/intc/v1.0/auth/result` | POST | 인증 결과(암호문) 조회 |

**토큰 관리:**
- 발급 시 10분 유효기간 설정 (`DATE_ADD(now(), INTERVAL 10 MINUTE)`)
- `ticket`, `iterators` 값은 토큰 발급 시 함께 수신 → 결과 복호화에 사용
- 매일 03:00 배치로 만료 토큰 일괄 삭제 (`cleanupExpiredTokens`)

### 2.3 암호화 방식

#### A. NICE 인증 결과 복호화 (NICE → AllconeApp)

NICE 서버에서 수신한 `enc_data`를 복호화하는 과정:

```
1. 키 유도 (PBKDF2)
   - Input: ticket(password), transactionId(salt), iterators(반복 횟수)
   - Algorithm: PBKDF2WithHmacSHA256
   - Output: 512bit → Base64Url 인코딩 (패딩 없음)
     * 앞 32바이트 → AES-256 키
     * 48~80바이트 → HMAC-SHA256 키

2. 무결성 검증
   - HMAC-SHA256(enc_data, hmacKey) → Base64Url (패딩 없음)
   - NICE에서 전달한 integrity_value와 비교

3. 복호화 (AES/GCM/NoPadding)
   - Base64Url 디코딩
   - 앞 16바이트 = IV (초기화 벡터)
   - 나머지 = 암호문
   - 128bit 인증 태그
   - 복호화 결과: JSON (이름, 전화번호, 생년월일, 성별, CI 등)
```

#### B. CI 값 양방향 암호화 (DB 저장용) - `CiCryptoService`

| 항목 | 내용 |
|------|------|
| **알고리즘** | AES-256-GCM |
| **키 유도** | PBKDF2WithHmacSHA256 (65,536 iterations) |
| **IV** | 12바이트 랜덤 (매 암호화 시 새로 생성) |
| **인증 태그** | 128bit |
| **저장 포맷** | Base64(IV[12bytes] + CipherText + Tag) |
| **설정값** | `ci.crypto.secret-passphrase`, `ci.crypto.salt` |
| **호환성** | 포털 웹 백엔드와 동일한 passphrase/salt 사용 |
| **저장 위치** | `tb_nice_cert.CI_VAL`, `tb_member.CI_VAL` |

#### C. CI 값 단방향 해시 (조회/매칭용) - `CiHashService`

| 항목 | 내용 |
|------|------|
| **알고리즘** | HmacSHA256 |
| **키** | `ci.crypto.hash-salt` (설정 파일) |
| **출력** | Base64 인코딩 (44자) |
| **특성** | 동일 CI + 동일 salt = 동일 해시 (결정적) |
| **용도** | 회원 조회, 중복 가입 방지, 아이디 찾기 |
| **저장 위치** | `tb_nice_cert.CI_HASH`, `tb_member.CI_HASH` |

> **참고**: 기존에 `NiceAuthService` 내에서 단순 SHA-256으로 해시하던 방식을 HmacSHA256으로 변경하였습니다. 포털 웹 팀에서 제공한 `CiHashService`를 그대로 사용합니다.

### 2.5 인증 목적 (Purpose)

| 코드 | 용도 | 후속 처리 |
|------|------|---------|
| `ALLCONE_APP_JOIN` | 회원가입 | certId → 회원정보 세팅 → tb_member INSERT |

- 접두사 `ALLCONE_APP_`로 포털 웹의 인증 기록과 구분 (VARCHAR(30) 이내)

## 3. CI 값 암호화/해시 체계

### 암호화 적용 흐름

```
[NICE 인증 결과 수신]
       |
       v
  평문 CI 값 (연계정보)
       |
       +-- CiCryptoService.encrypt(ci) --> CI_VAL (양방향, 복호화 가능)
       |                                    저장: tb_nice_cert, tb_member
       |
       +-- CiHashService.hash(ci) -------> CI_HASH (단방향, 조회용)
                                            저장: tb_nice_cert, tb_member
```

### 사용 시나리오

| 시나리오 | 사용 값 | 설명 |
|---------|---------|------|
| 아이디 찾기 | CI_HASH | certId로 CI_HASH 조회 → tb_member.CI_HASH와 매칭 |
| 비밀번호 재설정 | CI_HASH | 본인 확인 (CI_HASH 일치 여부) |
| 회원가입 | CI_VAL, CI_HASH | 둘 다 tb_member에 저장 |
| 인증정보 갱신 | CI_VAL, CI_HASH | 기존 회원의 CI 값 업데이트 |
| CI 복호화 필요 시 | CI_VAL | CiCryptoService.decrypt()로 평문 복원 |

### 포털 웹과의 호환

- **동일한 passphrase/salt/hash-salt** 사용 → 양쪽에서 암복호화 호환
- **CiCryptoService, CiHashService**: 포털 웹 팀에서 제공한 코드를 그대로 사용
- 기존 회원 데이터의 CI_HASH 재생성: **포털 웹 배치 작업으로 처리 예정**

### 4.4 로직 변경 사항

#### 로그인 실패 카운트 처리 변경

**변경 전 (tb_a_user_login_err_i):**
```sql
-- 별도 테이블에 INSERT ... ON DUPLICATE KEY UPDATE
INSERT INTO chainportaldb.tb_a_user_login_err_i
  (USER_ID, LOGIN_ERR_CNT, LOCK_YN, ...)
VALUES (#{userId}, 1, 'N', ...)
ON DUPLICATE KEY UPDATE
  LOGIN_ERR_CNT = LOGIN_ERR_CNT + 1,
  LOCK_YN = CASE WHEN LOGIN_ERR_CNT > 4 THEN 'Y' ELSE 'N' END
```

**변경 후 (tb_member 컬럼):**
```sql
-- tb_member 직접 UPDATE
UPDATE chainportaldb.tb_member
SET LGIN_FAIL_CNT = LGIN_FAIL_CNT + 1,
    LCK_YN = CASE WHEN LGIN_FAIL_CNT > 4 THEN 'Y' ELSE 'N' END,
    LCK_DT = CASE WHEN LGIN_FAIL_CNT > 4 THEN now() ELSE LCK_DT END
WHERE LGN_ID = #{userId}
```

#### 회원 탈퇴 처리 변경

**변경 후 (tb_member):**
- `MBR_NM`, `LGN_ID`, `PSWD`, `MBR_EML_ADDR`, `MBR_TELNO` → 익명화 (`'탈퇴회원'`)
- `CI_VAL`, `CI_HASH` → `'탈퇴회원'`으로 덮어쓰기
- `BRDT`, `MBR_EML_ADDR_PLAIN` → NULL
- 모든 동의 항목 → `'N'`
- `WHDWL_DT` → `now()`
- `LGN_ID` → `'탈퇴회원_' + LEFT(MBR_ID, 8)`로 변경 (재가입 대비)

#### 비밀번호 복잡도 검증 (신규)

- 8자 이상
- 영문, 숫자, 특수문자 중 2가지 이상 조합

#### 아이디 찾기 결과 마스킹 (신규)

- 6자 이상: 앞 3자 + `****` + 뒤 2자 (예: `abc****de`)
- 6자 미만: 앞 2자 + `**` (예: `ab**`)

### ISS-002: 로그인 실패 카운트 테이블 통합

**상태**: 해결  
- `tb_a_user_login_err_i` 별도 테이블 → `tb_member` 컬럼으로 통합
- CASE WHEN 로직으로 5회 초과 시 자동 잠금

### ISS-003: MBR_ID 생성 규칙

**상태**: 해결  
- 랜덤 6자리 + YYMM 형식
- DB 중복 체크 (최대 5회 재시도)
- 기존 `tb_a_user_m`에는 USER_ID가 PK였으나, `tb_member`에서는 MBR_ID가 PK, LGN_ID는 유니크 컬럼

### ISS-004: 회원 탈퇴 시 개인정보 처리

**상태**: 해결  
- 익명화 방식: `'탈퇴회원'` 문자열로 덮어쓰기
- CI_VAL, CI_HASH도 `'탈퇴회원'`으로 변경 (복원 불가)
- LGN_ID는 `'탈퇴회원_' + MBR_ID 앞 8자`로 변경하여 재가입 가능하도록 처리
